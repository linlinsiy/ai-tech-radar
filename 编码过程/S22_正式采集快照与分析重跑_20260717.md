# S22 正式采集快照与分析重跑

日期：2026-07-17  
范围：AWS 正式采集分析主流程、内部受控导入、设计文档与测试

## 目标

完整执行一次正式采集分析后，可复用本次已读取的正文重新执行 L2、L3 和后续简报，不重复访问外部网站，也不受 AWS 已处理缓存影响。

## 实现

- 正式任务完成 L2/L3 后，在 AWS 本地保存 data/analysis_snapshots/latest.json，并保留按批次命名的副本。
- 快照只保存已读取正文并完成 L2 尝试的文章；未读取正文的延迟候选不在快照内，避免把标题级信息误用于正式评分。
- 通过既有正式采集入口传入 scope=rerun 时，主流程读取最近快照，跳过 L0、站内发现、历史去重、标题路由和候选限额，直接重新执行 L2/L3。
- 重分析使用新批次号和 task_type=reanalysis 导入内部服务。相同 URL 的文章分析按新 Prompt 版本覆盖。
- 内部导入收到重分析替代标记后，将此次重分析范围内未重新生成的旧深度洞察标记为 superseded；L4 只查询 success 洞察，因此旧洞察不会混入新简报。

## 使用边界

- 首次部署该功能前的历史正式任务没有快照，不能使用 rerun；需重新采集一次，或另行开发基于 MySQL 原文的迁移工具。
- 快照为 AWS 单节点本地文件，应随数据目录一起备份；清理 data/analysis_snapshots 会失去重跑能力。
- 重跑后再调用内部简报生成接口，才会使用新的 L2/L3 结果。
