# S13 采集覆盖与简报均衡选题改造

日期：2026-07-15

## 背景

第一版工程已支持 RSS 与配置化 HTML 采集，但真实报告暴露出三个问题：

- 部分目标网站 RSS 不稳定或普通 HTML 无法取得正文，分类覆盖量不稳定。
- 某个可采集网站文章基数较大时，简报容易被该来源的立场和主题偏好主导。
- 原简报直接按 `value_score` 取固定 Top 15，技术深度较高的论文容易挤占工程、平台、产品和行业资讯。

本次改造保持既有 HTTP、XXL-Job、crontab 触发方式以及 AWS 到内部侧导入协议不变，将“文章自身价值判断”和“报告层面的代表性选择”拆开处理。

## 设计结论

1. 固定数据源仍是主采集入口，继续使用 RSS 与 HTML 混合采集。
2. 不在 L2 前按来源限制分析数量。采集成功并完成去重的文章全部进入 L2，避免按时间截断而漏掉重大事件。
3. 普通 HTML 请求失败、正文过短或页面为 JavaScript 空壳时，自动使用 Playwright/Chromium 降级抓取。
4. 分类覆盖不足或来源过度集中时，可调用搜索服务发现新 URL；第一阶段只接受已经配置的域名。
5. L2 五维综合分仍评价单篇文章；新增 L4 报告排序分和均衡选题，不改写原评分。
6. L4 以主题而非单篇文章为单位，同事件最多聚合 3 篇来源，主要事件可以越过来源软上限。
7. L5 使用固定文章结构，按一级分类组织自然段，技术和非技术资讯不再输出纪要式字段标题。
8. 采集审计和选题审计先保存本地 JSON，不新增 MySQL 表或业务字段。

## AWS 侧改动

### 1. 浏览器降级

新增 `tech-research/aws/app/crawler/browser_fetcher.py`：

- 使用 Playwright 同步 API 启动无头 Chromium。
- 单节点内使用锁串行抓取，避免浏览器并发占满内存。
- 缺少 Python 包或 Chromium 时只记录告警，不阻止服务启动。

修改 `web_crawler.py`：

- 普通 `requests` 请求仍为第一选择。
- 请求失败、可见文本少于配置阈值或命中 JavaScript 空壳特征时触发浏览器。
- 新增 `fetch_urls()`，让自动发现的详情 URL 复用现有正文解析和 `RawArticle` 数据模型。

修改 `deep/insight.py`：L3 全文请求失败时复用同一浏览器降级能力。

### 2. 自动 URL 发现

新增 `tech-research/aws/app/crawler/discovery_crawler.py`：

- 按一级分类生成中英文检索词。
- 兼容 `webPages.value`、`results`、`items` 三类常见搜索响应结构。
- 只保留域名与 `data_sources` 匹配的搜索结果。
- 使用 `WebCrawler.fetch_urls()` 抓取正文，再进入原日期过滤、去重和 L2 流程。

修改 `collect_job.py`：

- 固定源完成 L2 后计算分类覆盖和来源集中度。
- 仅在达到触发条件时执行一轮搜索补充。
- 自动发现文章不走旁路，仍执行 L1、L2、L3 和受控导入。

自动发现依赖企业可用搜索服务。当前没有确定服务地址，因此 `discovery.enabled=false`；代码和配置项已准备好，待提供 `search_endpoint`、查询参数和 `SEARCH_API_KEY` 后启用。

### 3. L2 元数据评分

修改 `processor/l2_analysis.py` 和 L2 Prompt：

- L2 输入由列表摘要扩展为摘要加有限正文样本。
- 增加来源名称、类型、域名、默认分类、采集方式、发布时间、抓取时间和内容长度。
- 来源可信度由来源类型或单源配置确定，不采用模型自行猜测的分数。
- 时效性由发布时间和抓取时间差确定，发布时间未知时使用中性分。
- L2 Prompt 版本升级到 `v2.7`。

综合价值分仍为技术深度、工程参考价值、趋势重要性、来源可信度和时效性的五项平均。

### 4. L3 门槛

L3 默认门槛从 7.0 调整为 8.0，并尊重 L2 的 `need_deep_analysis=false`。L3 继续使用 300 秒超时、并发 1、重试 0。

### 5. 采集审计

新增 `processor/collection_audit.py`，输出：

- `data/collection_audit/{batch_no}.json`：来源采集量、文章分类、评分、保留/丢弃和阶段统计。
- `data/source_health.json`：来源最近状态、最近成功时间和连续失败或空结果次数。

原导入失败请求仍保存到 `data/failed_imports/`。

## 内部侧改动

### 1. 新增 L4 选题层

新增 `tech-research/internal/app/jobs/briefing_selector.py`。报告排序分为：

```text
工程参考价值 35% + 趋势重要性 25% + 来源可信度 15%
+ 时效性 15% + 技术深度 10%
```

默认目标与软约束：

| 项目 | 默认值 |
|---|---:|
| 周报主题数 | 8 |
| 月报主题数 | 14 |
| 季报主题数 | 18 |
| 专题主题数 | 12 |
| 单一主来源主题数 | 最多 2 |
| 单一主来源比例 | 20% |
| 单一一级分类比例 | 35% |
| 工程类目标比例 | 55% |
| 研究类目标比例 | 25% |
| 同主题关联文章 | 最多 3 |

模型发布、产品发布、开源项目和政策监管类资讯，在趋势、可信度和综合价值达到阈值时标记为主要事件，可越过来源或分类软上限。若合格候选不足，最后一轮允许放宽软约束，避免机械凑数或生成空洞报告。

### 2. 简报任务接入 L4

重写 `internal/app/jobs/briefing_job.py` 的内部流程，但保留现有任务入口和响应字段：

- 移除 SQL 固定 `.limit(15)`，加载报告周期内全部成功 L2 候选。
- 查询来源信息和全部对应 L3 洞察。
- 先执行 L4 聚合与均衡选择，再把入选主题交给 L5。
- `related_article_ids` 和 `related_insight_ids` 保存真实 JSON 列表。
- 选题元数据保存到 `internal/data/briefing_selections/`。
- 内部大模型超时 300 秒，单次调用，不重试。

### 3. 固定简报 Prompt

更新 `internal/config/prompts/l5_briefing.properties`，版本为 `v2.0`。固定结构为：

- 标题和导语；
- 信息速览；
- 按现有一级分类顺序输出非空章节；
- 每个主题用 1 至 2 个自然段说明事实、问题、变化、方法和客观参考意义；
- 主题来源链接；
- 信号源索引。

Prompt 明确禁止角色分栏、行动清单、风险清单、投票区和纪要式三段标题。非技术资讯按产品、生态、市场、监管或业务变化自然表述。

## Linux 浏览器依赖安装

在 AWS 单节点执行：

```bash
cd /app/001804/aws
source venv/bin/activate
pip install -r requirements.txt

export PLAYWRIGHT_BROWSERS_PATH=/app/001804/aws/.playwright-browsers
python -m playwright install chromium
```

若缺少 Chromium 系统库，由有系统权限的运维人员执行：

```bash
python -m playwright install-deps chromium
```

安装后确认 `.playwright-browsers` 目录对 `appadmin` 可读和可执行，再重启现有 systemd 服务。
