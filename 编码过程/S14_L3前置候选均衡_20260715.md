# S14 L3前置候选均衡

## 背景

固定源采集和 L2 评分完成后，高产来源可能凭文章基数产生大量高分文章。原流程直接对全部高分文章执行 L3，再由内部 L4 均衡最终简报，虽然报告仍可控制来源分布，但会造成 L3 深度分析结果和调用资源向少数来源集中。

本次将来源均衡前移到 L2 与 L3 之间。所有合格 L2 仍然导入，不在采集或 L2 前截断；只控制哪些代表文章进入高成本 L3。

## 设计决定

- 保留 L4。L4 还负责按周报、月报、季报时间范围聚合主题、计算报告排序分和控制主题数量，不能由单批 L3 候选选择替代。
- L3 前置均衡承担主要来源控制；L4 调整为报告期编排和最终兜底。
- 同一分类内按标题、关键词、技术标签和机构名称聚合相似主题，只分析评分最高的代表文章。
- 常规候选按来源轮询，单来源最多 3 篇；单批最多 36 篇，单分类常规最多 35%。
- 模型发布、产品发布、开源项目和政策监管在趋势分不低于 9、可信度不低于 8、综合分不低于 7 时，可使用最多 6 个重大事件豁免名额。
- 分类素材不足时允许放宽分类上限，但来源上限不因补足数量而放宽；重大事件豁免除外。

## 代码改动

- 新增 `aws/app/deep/candidate_selector.py`，实现基础资格筛选、主题聚合、来源轮询、分类控制和重大事件豁免。
- `CollectOrchestrator` 在 L3 前调用候选选择器；L3 只处理选中结果，全部 L2 结果继续组装导入。
- 批次统计新增 `L3_selection`，记录基础合格数、主题数、选中数、排除数、来源/分类分布和候选明细。
- 采集审计文章增加 `l3_candidate` 和 `l3_success`，用于核对候选与实际成功情况。
- `config.properties` 新增 `l3_selection.*`，不改变 HTTP、XXL-Job、crontab 参数和启动方式。

## 默认配置

```properties
l3_selection.enabled=true
l3_selection.max_candidates_per_batch=36
l3_selection.max_candidates_per_source=3
l3_selection.max_source_ratio=0.20
l3_selection.max_category_ratio=0.35
l3_selection.topic_similarity_threshold=0.34
l3_selection.max_major_event_exceptions=6
```

## 运行影响

当前已经启动的 Python 进程不会热加载新增模块，正在执行的任务仍按旧逻辑运行。代码发布后需重启 AWS 服务；下一批任务开始应用候选均衡。数据库结构、导入协议和内部简报接口均未修改。

## 验证结果

- AWS Python 编译检查通过。
- AWS 采集与候选规则测试 13 项通过，新增覆盖来源轮询、同主题跨来源聚合和重大事件受限豁免。
- 内部 L4 回归测试 4 项通过，确认报告期主题编排、来源兜底和重大事件规则未受影响。
- 真实配置成功解析：单批 36、单来源 3、来源比例 20%、分类比例 35%、相似度 0.34、重大事件豁免 6。
