# S17 统一评分与全链路设计对齐

日期：2026-07-16  
范围：设计文档、AWS 采集分析链路、内部 L4/L5、数据库与 Prompt  
限制：未修改接口文档、接口入参、调度参数、服务启动方式和验收文档。

## 1. 改造原因

代码审计发现当前实现与最新确认方案仍有四类偏差：

1. L2、L3、L4 分别使用 `value_score`、`l3_rank_score`、`report_rank_score`，会在不同阶段重复解释文章价值。
2. L3/L4 保留重大事件标记、来源配额绕过和容量扩张，违背“所有资讯使用同一规则”的最新设计。
3. L4 可以从只有 L2 的文章中选题，简报可能缺少足够的深度材料。
4. L5 一次生成整篇报告，月报和季报存在输出压缩、主题遗漏且无法确定性校验的问题。

## 2. 最终处理规则

### 2.1 L1 候选路由

- 固定 AI 垂直来源沿用来源规则；混合来源增加批量标题语义路由。
- 路由只判断 `ai_related`、一级分类、资讯类型提示和置信度，不判断是否“重大”。
- 高置信度非 AI 内容在进入正文读取前过滤；低置信度或模型失败时保留候选并回退规则，避免误删。
- 每源仍最多保留 100 个候选，首批最多读取 50 篇；分类不足时从剩余标题池补采。

### 2.2 L2 唯一排序分

程序统一计算：

```text
rank_score =
  engineering * 30%
  + org_relevance * 25%
  + trend * 30%
  + timeliness * 5%
  + tech_depth * 10%
```

- 来源可信度 `score_credibility` 只作为事实质量门禁，不进入加权排序。
- `value_score` 作为旧字段兼容，新批次写入与 `rank_score` 相同的值。
- L3 默认门槛调整为 `rank_score >= 6.5`。

### 2.3 L3 候选均衡

- 准入条件为统一排序分、L2 深度分析建议、正文可用性和可信度门禁。
- 先聚合同一主题，再按 `rank_score` 排序。
- 单批最多 36 个主题；工程、行业、学术来源的单源软上限分别为 15%、10%、5%，一级分类上限为 35%。
- 候选主题不超过容量时全部保留；来源或分类维度稀疏时只停用对应均衡维度。
- 不存在 top-3、重大事件、资讯类型或来源身份豁免；配额受限时允许少选，不绕过规则补满。
- 为每篇文章记录未准入、同主题合并、均衡排除、L3 成功或失败原因。

### 2.4 L4 与 L5

- L4 SQL 候选池只查询 L2 成功且 L3 成功的文章。
- L4 直接使用 `rank_score`，不再计算 `report_rank_score`，也不使用重大事件标记。
- 同主题聚合后执行来源 15% 和动态分类上限；成功 L3 不足时生成较短报告，不以 L2 摘要补位。
- L5 按一级分类分别调用模型生成正文，再单独生成导语。
- 信息速览、信号源索引和最终 Markdown 由程序确定性组装。
- 程序校验 L4、信息速览、正文主题标记、信号源索引四处数量一致，并校验每个主题的来源链接；失败内容不写入成功草稿。

## 3. 主要代码变更

为防止评审继续读取历史版本，`设计/` 下重复保存的 DDL、运行配置和 L2/L3 Prompt 已改为唯一生效文件指引；实际内容只在 `tech-research/` 对应实现目录维护。

AWS 侧：

- `app/processor/title_router.py`：新增混合来源标题语义路由。
- `config/prompts/title_route.properties`：新增独立路由 Prompt。
- `app/processor/candidate_pool.py`：去除重大事件补采，按语义分类和置信度组织候选。
- `app/processor/l2_analysis.py`：新增组织相关性并计算唯一 `rank_score`。
- `app/deep/candidate_selector.py`：重写 L3 聚合、来源角色配额和分类配额。
- `app/deep/insight.py`：返回文章级执行结果和失败原因。
- `app/jobs/collect_job.py`：正式与验证流程共用分阶段主逻辑，导入统一评分及运营审计结果。

内部侧：

- `app/jobs/briefing_selector.py`：去除独立报告评分和全部身份例外，只从成功 L3 材料按统一分数选题。
- `app/jobs/briefing_job.py`：按分类生成、确定性组装并在写草稿前完整校验。
- `app/jobs/briefing_render.py`：新增无数据库依赖的 Markdown 组装与完整性校验模块。
- `config/prompts/l5_briefing.properties`：升级为 V3.0 分类分段 Prompt。
- `app/api/import_api.py`、`app/db/models.py`、`app/kb/markdown_gen.py`：同步组织相关性、统一排序分和来源角色。

## 4. 数据库升级

全新建库直接执行更新后的：

```text
tech-research/sql/ai_radar_ddl.sql
```

从 V2.6 旧库升级时，在重启内部服务前执行：

```text
tech-research/sql/ai_radar_alter_v2_7.sql
```

迁移新增：

- `ai_radar_source.selection_role`
- `ai_radar_article_analysis.score_org_relevance`
- `ai_radar_article_analysis.rank_score`
- `idx_rank_score`

旧记录将 `value_score` 回填到 `rank_score`，只用于历史兼容；新批次由 L2 按新公式生成统一分数。

## 5. 验证结果

- AWS 单元回归：31 项通过。
- 内部 L4/L5 单元回归：8 项通过。
- AWS 与 internal 应用代码 `compileall` 通过。
- `git diff --check` 通过，仅有仓库既有的换行符提示。

关键回归断言包括：

- 来源可信度不参与统一分数加权。
- 低置信度标题路由不会误删候选。
- 高分模型发布不能绕过来源配额或扩张批次/报告容量。
- L4 输出中不存在 `report_rank_score`、`must_include` 或重大事件清单。
- L5 四处主题数量和来源链接不一致时校验失败。

## 6. 部署影响

本次不改变原有 HTTP、XXL-Job、crontab 调用方式。部署时需同步 AWS/internal 代码、两侧配置和 Prompt；旧数据库必须先执行 V2.7 迁移。由于新 L4 只使用成功 L3，历史仅有 L2 的文章不会进入新简报，这是预期行为。
