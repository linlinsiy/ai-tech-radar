# S25 指定采集批次重分析

**日期**：2026-07-18  
**范围**：AWS 正式采集快照、`scope=rerun` 重分析和快照查询  
**不涉及**：重新采集、数据库表结构、内部导入协议、L3/L4/L5 选题规则

## 1. 背景

周度采集会持续更新最近快照，但月度采集结果需要保留，以便在周度评分规则调优完成后复用原始月度正文重新执行 L2/L3，避免重新访问外部来源、重新等待采集及受网络波动影响。

## 2. 设计

1. 每次正式采集都已保存 `data/analysis_snapshots/{IMP批次号}.json`；该文件是可回放资产，后续采集只会更新 `latest.json`，不会覆盖已有具名快照。
2. `scope=rerun` 新增可选参数 `reanalysis_batch_no`：
   - 传入：读取指定 `IMP` 快照并从 L2 开始重分析；
   - 未传：兼容旧行为，读取 `latest.json`。
3. 新增只读 `GET /api/v1/jobs/analysis-snapshots`，返回可选择的正式采集快照元数据：批次号、创建时间、文章数、原始范围、时间范围和来源范围。
4. 每次回放仍产生新的 `RERUN-*` 内部导入批次。原月度 `IMP-*` 仅标识采集输入，不会被重分析覆盖。

## 3. 实现

- `CollectionSnapshotStore` 新增具名 `load(batch_no)`、安全批次号校验和 `list_snapshots()`；列表跳过 `latest.json`、临时文件与损坏快照。
- `CollectOrchestrator.run` 和 `handle_collect_job` 透传 `reanalysis_batch_no`，`_run_snapshot_reanalysis` 按参数选择快照。
- 手动 HTTP 接口复用原 `POST /api/v1/jobs/collect` 请求模型；新增只读快照查询路由，不复制采集或分析逻辑。

## 4. 验证

- 具名快照可保存、列出、按批次读取，且非法路径批次号被拒绝。
- 指定重分析批次时调用具名读取方法；未指定时保留最近快照兼容路径。
- AWS 单元测试通过 43 项。
