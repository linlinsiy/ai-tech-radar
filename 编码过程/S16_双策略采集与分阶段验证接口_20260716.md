# S16 双策略采集与分阶段验证接口

日期：2026-07-16

## 背景

服务器探测建议给出了基于特定频道 URL 和 XPath 的适配方式，并提出使用媒体文章关键词匹配替代不可达的一手来源。复核发现部分 URL 已返回 404、部分 XPath 无匹配，且现有采集器只支持 CSS 选择器，无法直接验证该建议。

本次改造不修改正式 `/api/v1/jobs/collect`、XXL-Job、crontab、内部导入协议和接口验证文档。新增的验证入口将同一条正式流水线拆成采集、分析两个可停止的执行阶段，以同一组来源和时间范围对比两套策略；验证接口自身不实现采集、候选池、L2或L3业务逻辑。

## 设计结论

1. `server_recommended` 完整实现服务器建议涉及的频道 URL 和 XPath，严格模式下 XPath 无匹配即记录为空，不使用通用链接扫描掩盖配置错误。
2. `primary_resilient` 优先官方 RSS、重定向后的最终 RSS 和当前已验证入口；单个逻辑来源可以顺序尝试多个变体，首个取得候选的变体生效。
3. 服务器方案提出的“媒体关键词替代一手来源”仅记录为 `coverage_aliases`，用于统计其声称的覆盖范围，不篡改文章真实 `source_code`。
4. 采集接口不调用 LLM、不导入内部服务、不写正式 `processed_hashes.json`，将标题、链接、摘要和已取得的正文保存在本地 JSON。
5. 正式任务和验证任务共同调用 `CollectOrchestrator.collect_stage()` 与 `CollectOrchestrator.analyze_stage()`；验证服务只处理参数适配、本地JSON序列化和副作用选择。
6. 分析接口只读取指定采集批次，并进入共享候选池、L2、L3前置均衡和L3分析；验证模式禁止L3重新访问原文。
7. 原正式采集分析接口、请求参数、调度方式及部署启动方式保持不变。

## 共享主逻辑与副作用

`collect_stage()` 统一负责来源计划执行、爬虫创建、顺序回退、日期过滤、可选正文补取及来源级执行结果。生产使用 `configured` 计划；两个验证策略只改变来源计划，不改变采集执行器。

`analyze_stage()` 统一负责批内URL去重、候选池规划、正文充分性检查、内容去重、L2、分类补选、L3候选均衡和L3分析。调用方只能选择以下副作用，不得复制阶段内部逻辑：

| 执行方式 | 历史缓存读取 | 已处理缓存写入 | 自动发现 | L3远程补抓 | 内部导入 |
|---|---|---|---|---|---|
| 正式完整任务 | 开启 | 开启 | 开启 | 开启 | 开启 |
| 验证只采集 | 不进入分析 | 关闭 | 关闭 | 不执行 | 关闭 |
| 验证批次分析 | 关闭 | 关闭 | 关闭 | 关闭 | 关闭 |

## 两套策略

### server_recommended

- 对36氪、阿里云、CSDN、钛媒体、腾讯云、零壹财经、同花顺和开源中国应用服务器建议的URL与XPath。
- 对建议涉及的来源启用 `xpath_strict=true`，保证验证结果反映建议本身是否有效。
- 其余来源沿用当前配置。
- 关键词替代关系保存在 `coverage_aliases` 和 `coverage_alias_distribution` 中，不伪造文章来源。

### primary_resilient

- DeepMind 改为官方 `https://deepmind.google/blog/rss.xml`，失败后尝试官方 Blog 页面。
- HuggingFace 优先官方 Feed，失败后尝试官方 Blog 页面。
- LangChain 优先重定向后的 `https://www.langchain.com/blog/rss.xml`，保留旧地址为后备。
- 钛媒体和开源中国继续使用当前可用 RSS。
- 腾讯云优先当前可用 AI 专栏，失败后才使用开发者首页关键词粗筛。
- 36氪使用 AI 频道并按 `/p/数字` 文章路径约束，不再依赖标题必须出现 AI 关键词。
- 零壹财经、同花顺保留当前可用首页和现有包含/排除关键词。

## 代码改动

### 采集器

修改 `tech-research/aws/app/crawler/web_crawler.py`：

- 增加可选 `list_link_xpath`、`list_title_xpath` 和 `xpath_strict`。
- 增加 `article_url_pattern`，从宽泛页面中限定文章详情 URL。
- 增加来源级 `referer`。
- 修复 `list_selector` 直接选中 `<a>` 时无法识别链接的问题。
- 所有新配置默认关闭，不改变现有生产来源行为。

新增 `tech-research/aws/app/crawler/source_strategies.py`：

- 定义两套验证策略及来源变体顺序。
- 定义服务器方案中的关键词替代关系。
- 按正式数据源配置生成全量或指定来源的验证计划。

### 本地批次与分析

新增 `tech-research/aws/app/jobs/validation_job.py`：

- `ValidationCollectionService`：调用共享 `collect_stage()`，只补充本地批次统计和序列化。
- `ValidationAnalysisService`：读取本地批次后调用共享 `analyze_stage()`，不维护独立候选规划或分析流程。
- `ValidationStore`：使用临时文件加原子替换写入 JSON，并限制批次号字符，防止路径越界。

修改 `tech-research/aws/app/jobs/collect_job.py`：

- 提取 `collect_stage()`，正式任务和两种验证采集策略共用来源执行逻辑。
- 提取 `analyze_stage()`，正式任务和验证批次分析共用 L1/L2/L3 主逻辑。
- 历史去重、缓存写入、自动发现、L3远程补抓、审计和内部导入均由调用模式显式控制。

修改 `tech-research/aws/app/deep/insight.py`：

- 新增 `allow_content_fetch`，默认值为 `true`，正式流程不变。
- 验证分析将其设为 `false`，缺少已采集正文时不重新访问外部 URL。

### HTTP接口

修改 `jobs_api.py` 和 `main.py`，新增：

```text
POST /api/v1/validation/collect
POST /api/v1/validation/analyze
```

采集接口除 `strategy` 外，其余参数与 `/api/v1/jobs/collect` 一致：

```json
{
  "strategy": "primary_resilient",
  "scope": "sources",
  "sources": "oschina-ai,36kr-ai",
  "from_date": "2026-06-16",
  "to_date": "2026-07-16",
  "task_type": "manual_backfill"
}
```

返回的 `batch_no` 用于分析接口：

```json
{
  "collection_batch_no": "VAL-COL-20260716-120000-000000"
}
```

## 本地文件

```text
data/validation/collections/VAL-COL-*.json
data/validation/analyses/VAL-ANA-*.json
```

采集文件包含请求参数、来源每个变体的 HTTP 状态、最终重定向地址、错误信息、候选数量、最终变体、来源分布、正文可用数和原始文章。分析文件包含L2结果、丢弃结果、L3洞察、来源/分类分布和服务器替代标签统计。

这些文件不会进入内部 MySQL，也不会影响正式采集去重缓存。

## 验证结果

新增测试覆盖：

- XPath 严格解析和无匹配行为。
- CSS选择器直接选中链接。
- URL路径规则。
- 两套来源策略差异。
- 服务器关键词替代标签不修改真实来源。
- 验证批次原子读写和批次号路径校验。
- 两个同步任务在线程池执行。
- 验证L3不会重新采集正文。
- 正式任务与验证任务均委托到共享采集、分析阶段。
- 验证分析关闭历史缓存读写、自动发现和远程正文补抓。
- 只采集模式不会初始化LLM客户端或分析器。

使用临时目录对 `oschina-ai` 进行真实网络冒烟验证：

```text

primary_resilient: HTTP 200, article_count=45, success_sources=1
```

前者使用服务器建议但已失效的 `/ai` 页面；后者使用现有 AI RSS。测试文件随临时目录删除，没有写入工程 `data`。

## 使用限制

- 全量验证会读取每个来源最多100条候选及其可取得的正文，WEB来源较多时执行时间较长，首次比较建议通过 `scope=sources` 指定同一组来源。
- `server_recommended` 的空结果是验证结论，不会自动切换为合理策略。
- `primary_resilient` 的后备端点只在前一变体没有候选时启用。
- 分析接口需要配置有效的外部模型参数；L2/L3失败仍会在分析批次统计中体现。
