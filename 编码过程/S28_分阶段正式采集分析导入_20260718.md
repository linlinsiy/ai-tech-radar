# S28 分阶段正式采集、分析与导入

**日期**：2026-07-18  
**范围**：既有 `/api/v1/validation/collect`、`/api/v1/validation/analyze` 及共享采集导入逻辑

## 1. 背景

原分阶段接口只用于验证：采集结果落在 `data/validation`，分析不导入 internal/MySQL，与正式 `collect` 全流程的落地和副作用不一致，无法支持“月度采集一次，再分别做周度和月度分析”的工作方式。

## 2. 改造结果

1. 保留两个既有接口路径，不新增验证接口。
2. 分阶段采集使用正式来源配置和共享采集阶段，完整读取正文，保存 `COL-*` 到 `data/analysis_snapshots/`，并写入来源审计；不执行 L2/L3、不导入、不写已处理去重缓存。
3. 所有快照使用同一 JSON 结构，不额外维护快照类型字段。`COL-*` 表示未分析的采集结果；`IMP-*`/`RERUN-*` 表示已分析结果，可再次重跑。
4. 分阶段分析读取 `COL-*` 时支持 `from_date/to_date` 和 `collection_period`，执行共享候选池、L2、自动发现、L3 和受控导入；读取 `IMP-*`/`RERUN-*` 时复用正式重分析路径。成功后写入 internal/MySQL，并保存或更新 `IMP-*` 分析快照和审计。
5. `COL-*` 可由分阶段分析或 `POST /api/v1/jobs/collect` 的 `scope=rerun` 消费；完整任务生成的 `IMP-*` 同样可由分阶段分析重跑，确保两种执行方式的文件相互可用。
6. 分阶段分析和快照重分析共用 `CollectOrchestrator.import_analysis_results()` 生成受控导入请求；导入失败仍保存 `data/failed_imports/` 重试文件。
7. 内部导入遇到重复 URL 时统一更新 `article.import_batch_id` 到最近成功批次，修复“导入数量与最新批次查询数量不一致”。

## 3. 运行方式

- 月度采集：调用分阶段采集，取得 `COL-*` 批次。
- 周度验证分析：使用同一个 `COL-*` 加一周日期范围调用分阶段分析，结果会正式导入。
- 月度分析：使用同一个 `COL-*` 不传日期范围再次调用分阶段分析。
- 原 `POST /api/v1/jobs/collect` 仍保持一键全流程，也可以通过 `scope=rerun` 消费 `COL-*` 或既有分析快照。
