# S15 分阶段采集与运营统计

## 背景

上一版在全部文章完成 L2 后才进行 L3 来源均衡。高产来源仍会产生大量正文请求和 L2 结果，分类覆盖不足时也缺少从已发现标题中定向补读正文的机制。本次将候选控制前移，并增加 MySQL 批次运营统计，用于持续观察 L1、L2、L3 的来源和分类分布。

## 设计决定

- 每个来源最多保留 100 条标题、链接、时间和可用列表摘要；WEB 采集先解析列表，只有入选候选才请求详情页。
- 每源常规首批最多读取 50 篇正文。标题规则只做重大事件识别、分类预测和候选排序，不生成正式评分。
- 首批 L2 后检查分类覆盖，从延迟候选池为不足分类补读正文；每类常规最多 10 篇，疑似重大事件不受该上限限制。
- 正文不足 100 字符时不正式评分；URL 在候选阶段判重，正文在 L2 前计算内容指纹。只有 L2 成功后提交本地已处理状态，失败文章保留重试机会。
- L3 阈值由 8.0 调整为 7.0。普通候选继续执行来源和分类软限制；短周期来源或分类较少时只放松对应维度，不机械平均。
- 重大模型、产品、开源、政策以及重大工程提升先按同事件聚合，再绕过普通来源、分类和 36 篇目标上限，不再固定为最多 6 条。
- L3 Prompt 输出约 600 至 1200 个中文字符的深度洞察；L5 优先使用 L3 洞察成文，避免只改写 L2 摘要。

## 运营表

新增 `ai_radar_pipeline_operation`，每个 `batch_no` 一行，核心字段：

| 阶段 | 指标 |
|---|---|
| 批次 | `batch_no`、`batch_time`、`import_batch_id` |
| L1 | 候选总数、来源数量分布 |
| L2 | 筛选保留数、来源分布、一级分类分布 |
| L3 | 入选数、来源分布、一级分类分布 |
| 扩展 | `stage_detail` JSON，保存首批、补采、内容不足、重复和失败统计 |

AWS 不新增 MySQL 依赖。阶段指标作为现有受控导入请求的可选 `operation_metrics` 字段传递，由 internal 在同一事务中写入。旧调用方不传该字段时继续按原逻辑导入。

现有数据库升级执行：

```bash
mysql -u USER -p DATABASE < tech-research/sql/ai_radar_alter_v2_6.sql
```

升级脚本使用 `CREATE TABLE IF NOT EXISTS`，不会修改或清理已有文章、分析和简报数据。

## 代码改动

- `aws/app/crawler/base.py`：增加候选发现、候选正文读取的兼容接口。
- `aws/app/crawler/web_crawler.py`：列表发现与详情读取分离。
- `aws/app/processor/candidate_pool.py`：新增首批选择、分类预测、延迟补采和疑似重大事件保护。
- `aws/app/jobs/collect_job.py`：重排候选、正文、L2 补采、L3 和运营统计流程。
- `aws/app/processor/dedup.py`：L2 成功后批量提交 URL/内容指纹。
- `aws/app/deep/candidate_selector.py`：7 分门槛适配、短周期软限制和重大工程提升保护。
- `internal/app/api/import_api.py`、`internal/app/db/models.py`：接收并持久化运营统计。
- `tech-research/sql/ai_radar_ddl.sql`、`设计/ai_radar_ddl.sql`：同步完整表结构。
- `tech-research/sql/ai_radar_alter_v2_6.sql`：现有数据库增量升级脚本。

## 本地审计

- `data/collection_audit/{batch_no}.json`：记录来源候选数、是否触达候选上限和各阶段统计。
- `data/candidate_pool/{batch_no}.json`：记录延迟候选标题、链接、预测分类、重大事件标记和是否被补采。
- `data/failed_imports/{batch_no}.payload.json`：导入失败时同时保留运营统计，可使用原导入接口重试。
