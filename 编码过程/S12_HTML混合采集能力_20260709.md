# S12 HTML 混合采集能力

## 背景

国内科技媒体、金融科技机构、监管与产业动态页面普遍缺少稳定 RSS，原 RSS 优先采集方式无法覆盖“AI 在金融领域应用”“国内行业动态”“国内监管合规”等内容缺口。

本次改造目标是补充 HTML 爬虫能力，同时保持采集任务触发方式和后续数据处理链路稳定：

- 不新增 XXL-Job、crontab 或 HTTP 触发接口；
- 不改变 `RawArticle` 输出结构；
- 不调整内部 MySQL 表结构；
- RSS 源仍使用原 `RSSCrawler`，HTML 源通过配置选择 `WebCrawler`。

## 设计取舍

采集层采用 `fetch_method` 分流：

| 配置值 | 采集器 | 说明 |
|---|---|---|
| `rss` | `RSSCrawler` | 继续处理 RSS/Atom 源 |
| `web` / `html` | `WebCrawler` | 处理国内 HTML 页面 |
| `api` | `APICrawler` | 保留 API 接入扩展 |

HTML 爬虫采用“配置优先 + 通用规则兜底”的方式：

- 配置了 `list_selector`、`detail_content_selector` 等 CSS 选择器时，优先按站点配置解析；
- 未配置或页面结构变化时，扫描常见文章卡片、链接和详情正文区域；
- 列表阶段可使用 `include_keywords` / `exclude_keywords` 粗筛，最终分类和价值判断仍交给 L2。

## 代码改动

### 1. `WebCrawler` 通用化

文件：`tech-research/aws/app/crawler/web_crawler.py`

主要能力：

- 使用浏览器 User-Agent 请求 HTML 页面；
- 默认请求超时 20 秒，请求间隔 1 秒；
- 支持列表页、分页、详情页抓取；
- 支持标题、作者、发布时间、正文区域的可配置 CSS 选择器；
- 支持中文相对时间和常见绝对日期解析；
- 输出统一 `RawArticle`，`raw_summary` 保存正文摘要，`raw_html` 保存正文 HTML 片段。

### 2. 采集工厂支持 `html`

文件：`tech-research/aws/app/crawler/base.py`

`CrawlerFactory.create()` 增加 `html` 作为 `web` 的别名：

```python
elif method in ("web", "html"):
    from crawler.web_crawler import WebCrawler
    return WebCrawler(source)
```

### 3. 配置读取保留扩展字段

文件：`tech-research/aws/app/config.py`

`AWSConfig.get_data_sources()` 不再只返回固定字段，而是保留每个 `data_sources[N].*` 下的所有配置项，使 HTML 爬虫可以读取选择器、关键词、分页和限流参数。

### 4. AWS 数据源配置

文件：`tech-research/aws/config/config.properties`

调整为 RSS + HTML 混合采集：

- `hundsun-research`：改为 HTML 源，补充金融科技与 AI 应用；
- `thsh-fintech`：改为移动端 HTML 源，补充金融科技与 AI 应用；
- 新增 `qbitai-ai`：量子位 AI 资讯；
- 新增 `36kr-ai`：36 氪移动端 AI 相关资讯；
- `data_sources.count` 调整为 `20`；
- `security.allowed_domains` 增加 `m.10jqka.com.cn`、`www.qbitai.com`、`m.36kr.com`。

## 配置示例

```properties
data_sources[N].code=qbitai-ai
data_sources[N].name=量子位AI频道
data_sources[N].type=tech_media
data_sources[N].category=行业动态
data_sources[N].access_url=https://www.qbitai.com/
data_sources[N].domain=www.qbitai.com
data_sources[N].fetch_method=web
data_sources[N].list_selector=article|h4|.post|.item|li
data_sources[N].detail_content_selector=article|.content|.article-content|.post-content|main
data_sources[N].max_pages=2
data_sources[N].page_url_pattern=https://www.qbitai.com/page/{page}
data_sources[N].max_articles=20
data_sources[N].timeout_seconds=20
data_sources[N].request_interval_seconds=1
data_sources[N].enabled=true
```

多个 CSS 选择器使用 `|` 分隔，避免和 CSS 逗号语法冲突。

## 验证建议

单独验证 HTML 源：

```bash
curl -sS -X POST 'http://127.0.0.1:9003/api/v1/jobs/collect' \
  -H 'Content-Type: application/json' \
  -d '{"scope":"sources","sources":["qbitai-ai"],"task_type":"manual_backfill"}'
```

验证金融 AI 补充源：

```bash
curl -sS -X POST 'http://127.0.0.1:9003/api/v1/jobs/collect' \
  -H 'Content-Type: application/json' \
  -d '{"scope":"sources","sources":["hundsun-research","thsh-fintech"],"task_type":"manual_backfill"}'
```

观察点：

- AWS 日志中应出现 `开始 HTML 采集`；
- L0 采集数量应大于 0；
- L2 能正常输出分类、摘要、评分；
- 导入内部服务后，文章仍按原表结构入库。

## 后续注意事项

- 国内站点页面结构变化较快，若某站点 L0 采集数量持续为 0，优先调整该源的 CSS 选择器配置；
- 36 氪桌面端存在验证码，当前配置优先使用移动端页面；
- 新增数据源编码后，内部 `ai_radar_source` 主数据建议同步补齐，否则当前导入逻辑会使用默认来源兜底，文章可入库但来源统计不准确；
- 若未来 HTML 源数量明显增加，再考虑引入 Scrapy/Playwright、代理池、增量游标等中长期能力。
