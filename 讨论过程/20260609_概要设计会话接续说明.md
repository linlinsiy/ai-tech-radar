## 5. 已确认的架构决策

### 5.1 总体部署边界

已确认采用两个主要运行区：

| 区域 | 定位 | 主要职责 |
|---|---|---|
| 亚马逊云侧外部公开资讯处理区 | 面向公网资讯采集与公开内容分析 | 访问外部技术网站/RSS/API，完成采集、解析、去重、摘要、分类、深度分析、结构化结果生成 |
| 公司内部知识库使用区 | 面向内部知识沉淀、检索和问答 | 接收亚马逊云侧加工成果，写入内部数据库和知识库，基于内部大模型提供检索和 RAG 问答 |

已确认原则：

1. 亚马逊云侧作为外部采集与公开内容分析环境，符合公司资源和网络策略。
2. 可以建设“亚马逊云侧到公司内部受控导入接口”的导入链路。
3. 公司内部已有或可提供内部知识库、内部大模型、任务调度平台和数据库资源。
4. 亚马逊云侧只处理外部公开资讯，不处理公司内部业务数据。
5. 外部大模型只接收公开文章内容和分析提示词。
6. 公司员工问题、内部知识库内容、内部业务数据不发送至亚马逊云侧或外部大模型。
7. 内部检索、问答、知识复用全部在公司内部完成。

### 5.2 系统总体架构与模块划分

已确认系统按“目标态架构、首期轻量实现”的思路设计。

建议的业务层包括：

| 层级 | 部署位置 | 主要模块 |
|---|---|---|
| 数据源接入层 | 亚马逊云侧 | 数据源配置、白名单访问、RSS/API/网页采集 |
| 外部内容处理层 | 亚马逊云侧 | 内容解析、去重、有效性检查、外部大模型分析编排 |
| 结构化成果生成层 | 亚马逊云侧 | 文章记录、分析结果、报告草稿、任务元数据打包 |
| 受控导入层 | 公司内部侧 | 内部受控导入接口、数据校验、入库适配 |
| 内部知识服务层 | 公司内部侧 | 业务数据库、内部知识库、内部大模型、RAG 问答 |
| 应用输出层 | 公司内部侧 | 知识库检索入口、智能问答、周报/月报/专题报告草稿 |
| 运行治理能力 | 两侧均涉及 | 任务调度、日志、失败重试、数据源健康检查等 |

用户已确认：

1. 首期可以不建设复杂前端管理后台。
2. 公司内部可以提供“内部应用节点”，承接受控导入、数据库写入和知识库入库。
3. 任务调度平台为公司公用服务，在亚马逊云可以使用其定时调度功能，不需要关心其实际部署位置。

因此后续文档中不要写成“内部侧调度平台必须部署在内部并主动触发”，应写为“公司公用任务调度服务可触发亚马逊云侧任务，或由亚马逊云侧接入该调度能力”。

### 5.3 数据流转与接口链路

已确认主链路包括：

| 链路 | 流向 | 说明 |
|---|---|---|
| 任务触发链路 | 公司公用任务调度服务到亚马逊云采集分析服务 | 按定时策略触发采集任务，支持补跑和范围参数 |
| 外部采集分析链路 | 亚马逊云采集分析服务到外部资讯源/外部大模型 | 访问公开网站、RSS、API，调用外部大模型分析公开文章 |
| 受控导入链路 | 亚马逊云采集分析服务到公司内部受控导入接口 | 将结构化成果、来源链接、任务元数据导入内部应用节点 |
| 内部知识使用链路 | 内部用户到内部知识库/内部大模型 | 内部检索、问答、简报复用均在公司内部完成 |

用户已确认和修正：

1. 亚马逊云侧只能通过受控导入接口进入内部，不能直连内部数据库。
2. 受控导入接口可以由内部应用节点提供。
3. 日志不再接入公司 APM，见后续新增限制。
4. 任务状态可由调度平台监控，无需单独建设任务状态中心。

### 5.4 新增部署限制

用户在 2026-06-03 新增限制：

1. 亚马逊云部署仅按照单节点部署方式。
2. 日志存储本地。
3. 不对接 APM。

后续概要设计必须使用该最新口径。此前关于 APM SDK/接口接入、多节点、Redis 分布式锁等内容应作为被覆盖的历史讨论，不应写入最终概要设计主方案。

## 6. 亚马逊云主机部署组件口径

用户曾澄清：其询问的不是业务内部功能拆分，而是支撑服务启动的基础组件，例如 Java、MySQL、本地缓存等。

按照最新单节点和本地日志限制，建议后续概要设计中将亚马逊云主机组件写为：

| 组件 | 建议 | 说明 |
|---|---|---|
| Linux 操作系统 | 必需 | 作为服务运行基础，使用公司认可的发行版 |
| 服务运行时 | 必需 | 若服务采用 Java/Spring Boot，则部署 JDK；若复用现有 Python 流水线，则部署 Python 环境 |
| 采集分析服务程序 | 必需 | 承载采集、解析、去重、模型调用、结构化结果生成、受控导入客户端等内部功能 |
| 进程管理 | 必需 | 可用 `systemd` 或公司标准进程托管方式 |
| 本地临时目录 | 必需 | 保存采集临时文件、中间结果、导入包缓存和失败重试数据 |
| 本地日志目录 | 必需 | 保存运行日志、错误日志和接口调用日志；不接入 APM |
| 本地缓存 | 建议 | 单节点场景可用内存缓存或本地文件缓存，不必引入 Redis |
| Nginx/反向代理 | 可选 | 如需暴露 HTTP 任务触发、健康检查或统一 TLS 入口可部署 |
| 本地数据库 | 原则上不建议作为必需 | 亚马逊云侧不承载正式业务数据；如需短期持久化重试队列，可评估轻量本地库，但首期不应强依赖 |
| Redis | 不建议首期部署 | 单节点场景无需分布式锁和跨节点队列 |
| 安全凭证配置 | 必需 | 管理外部大模型 API Key、受控导入接口凭证、证书等 |
| 网络访问配置 | 必需 | 外部网站白名单、外部大模型 API 访问、内部受控导入接口访问 |
| 调度平台接入能力 | 必需 | 接入公司公用任务调度服务的定时调度能力 |

## 7. 数据对象与存储设计已确认内容

设计项 4 讨论中，用户已确认：

1. 亚马逊云侧允许短期保存采集和分析中间结果，用于失败重试和排障。
2. 公司内部业务数据库可以承接结构化对象，但需要有明确 MySQL 表设计。
3. 公司知识库可以定义知识库标签 schema，用于标记每个文件的扩展信息，如标题、作者、关键词等。

针对第 3 点已形成设计理解：

1. “文章摘要卡片 / 深度洞察卡片 / 简报文档”指知识库入库内容粒度。
2. “知识库标签 schema”用于承载这些内容的扩展元数据。
3. 如果知识库以文件为单位入库，可将每篇文章摘要、每条深度洞察、每份简报组织成独立 Markdown/文本文件，并通过标签 schema 标记标题、作者、来源、关键词、分类、评分、发布时间、导入批次等信息。

### 8.7 初版约束建议

1. `ai_radar_article.url_hash` 建唯一索引，避免同一 URL 重复入库。
2. `ai_radar_source.source_code` 建唯一索引。
3. `ai_radar_import_batch.batch_no` 建唯一索引。
4. `article_analysis.article_id` 与 `deep_insight.article_id` 建索引。
5. 暂不在 MySQL 存储完整原文全文，除非后续明确知识库或合规允许。
6. 如后续知识库可返回文件 ID，可补一张 `ai_radar_kb_mapping` 表，记录文章、洞察、简报与知识库文件的映射关系。
