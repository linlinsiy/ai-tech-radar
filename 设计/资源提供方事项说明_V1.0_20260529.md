# AI技术趋势雷达资源提供方事项说明

## 1. 背景与目标

AI技术趋势雷达拟在公司内部环境建设外部AI技术资讯采集、分析与知识复用能力。系统持续跟踪公开技术媒体、厂商博客、学术平台和开源社区内容，完成采集、解析、去重、摘要、分类、深度分析和结构化输出，并将加工成果沉淀到公司内部知识库，供检索、问答、技术简报和方案调研使用。

本事项需资源提供方支持可访问外部公开技术站点的亚马逊云采集分析环境、外部大模型调用资源、公司内部受控导入接口、内部数据库、内部知识库、内部大模型和任务调度等配套资源。总体原则是：外部公开资讯可在亚马逊云侧采集和分析；公司业务数据、员工问题和内部知识库内容不发送至亚马逊云侧或外部大模型；内部检索与RAG问答使用公司内部大模型完成。

## 2. 总体架构与部署方案

【图1：总体架构图，见 [总体架构图.mmd](总体架构图.mmd)】

系统分为三个区域：

| 区域 | 主要能力 | 需提供或确认的资源 |
|---|---|---|
| 外部公网资源 | 外部公开资讯源、外部大模型服务 | 目标网站/RSS/API/公众号公开内容访问白名单，外部大模型API资源 |
| 亚马逊云侧：外部公开资讯处理区 | 采集、解析、去重、外部模型编排、结构化结果生成、运行日志和任务状态记录 | 亚马逊云采集分析节点，出网访问权限，模型调用网络权限，任务日志与状态留存能力 |
| 公司内部：知识库使用区 | 任务触发、受控导入、数据写入、知识库沉淀、内部检索与RAG问答 | 公司任务调度平台、内部应用节点、受控导入接口、业务数据库、内部知识库、内部大模型 |

公司任务调度平台触发亚马逊云侧采集任务并获取状态；亚马逊云侧仅处理外部公开资讯，生成结构化成果后通过企业受控导入链路进入内部应用节点；内部应用节点负责写入业务数据库和内部知识库，并调用内部大模型支撑内部用户检索与问答。

## 3. 数据流与安全边界

【图2：数据流图，见 [数据流图.mmd](数据流图.mmd)】

数据流转口径如下：

| 流向 | 数据内容 | 说明 |
|---|---|---|
| 外部公开资讯源 → 亚马逊云侧 | 公开标题、链接、发布时间、摘要或正文 | 按白名单访问公开网站、RSS或API |
| 亚马逊云侧 → 外部大模型 | 公开文章内容、分析提示词 | 用于摘要、分类、关键信息抽取、深度分析和报告草稿生成 |
| 外部大模型 → 亚马逊云侧 | 摘要、分类、分析结果、报告草稿 | 返回公开资讯加工结果 |
| 亚马逊云侧 → 公司内部受控导入接口 | 结构化成果、来源链接、任务元数据 | 进入内部数据库和内部知识库 |
| 内部用户/技术团队 → 内部知识库/内部大模型 | 内部问题、检索请求、知识库召回内容 | 全部在公司内部完成，回答应带来源引用 |

安全边界要求：公司业务数据、员工问题、内部知识库内容不得发送到亚马逊云侧或外部大模型；外部大模型只处理外部公开文章内容；进入公司内部的是公开资讯加工成果及任务元数据。

## 4. 处理逻辑

【图3：处理逻辑图，见 [处理逻辑图.mmd](处理逻辑图.mmd)】

处理链路为：任务调度触发定时任务或手动补跑；读取数据源配置，包括访问域名、采集频率和启用状态；通过RSS、API或网页方式采集外部公开资讯；采集成功后解析标题、链接、发布时间、摘要或正文；按URL、标题、内容指纹和抓取状态进行去重与有效性检查；对有效文章进行基础分析，生成摘要、分类和关键信息；高价值内容触发深度分析，提取技术背景、实现思路、影响和参考价值；最终生成文章记录、分析结果、报告草稿和任务元数据，调用内部受控导入接口写入内部数据库和内部知识库。采集失败时记录失败原因，用于重试、替换数据源或排障。
