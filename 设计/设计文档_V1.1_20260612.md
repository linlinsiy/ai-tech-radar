## 1. 设计概述

### 1.1 建设目标

AI技术趋势雷达面向公司及部门内部读者，持续收集公开AI资讯，形成可追溯、可检索、可阅读的技术趋势内容。系统重点发挥“雷达”作用：

- 广泛覆盖模型、Agent、多模态、基础设施、应用、治理、生态、行业及金融应用等方向。
- 客观呈现本期发生了什么、解决什么问题、采用什么方法及其适用边界。
- 兼顾工程实践、研究进展、产品和行业信号，避免被单一来源或单一内容类型主导。
- 保存原始来源、分析过程和简报引用关系，支持核验、知识检索和后续复用。

简报不按领导、业务、技术等角色拆分章节，不代替读者做行动决策，也不输出行动清单或风险清单。不同角色从同一份客观报告中获取方向、方案、技术和业务结合方面的参考。

### 1.2 本次确认的核心设计

1. 系统继续采用亚马逊云侧与公司内部侧分区部署，外部公开数据只按受控方向导入内部。
2. 固定数据源支持 RSS 与 HTML 混合采集；普通请求失效或页面为 JavaScript 空壳时，自动使用 Chromium/Playwright 降级抓取。
3. 每个来源先形成最多 100 条标题/链接候选；混排来源使用轻量语义分类批量判断 AI 相关性、预测方向和资讯类型提示，关键词规则保留为失败降级。首批最多读取 50 篇正文进入 L2，剩余候选延迟保存并按覆盖缺口补读。
4. 当分类覆盖不足或来源过度集中时，系统通过 robots.txt、Sitemap 和受限同域遍历自动发现新 URL；不要求人工判断重要来源、补充 URL、搜索地址或 API Key。
5. L2 使用券商技术岗位领域匹配度、趋势重要性、工程参考价值、技术深度和时效性计算唯一 `rank_score`。领域匹配度采用固定六档，重点覆盖证券业务 AI、软件研发与 AI 工程、必须了解的重大 AI 事件；来源身份不形成分数或门禁。
6. L3 与 L4 使用同一个 `rank_score`。L3 在同事件合并后按分数取前 50 篇生成深度材料；L4 只从成功 L3 主题中按分数完成跨周期聚合和篇幅控制，来源与分类分布只观察、不干预选题。
7. L5 按分类分段生成并组装固定结构，全部入选主题必须同时出现在信息速览、正文和信号源索引。技术内容优先使用 L3 机制、数据和边界；非技术内容按类型展开事实，不强行编造技术方案。
8. 不改变既有 HTTP、XXL-Job、crontab 触发方式；批次运营统计补充语义筛选、L3 失败原因、最终洞察覆盖率和 L5 完整性校验结果。

### 1.3 设计原则

| 原则 | 设计含义 |
|---|---|
| 外部采集、内部沉淀 | 亚马逊云负责公开资讯采集与分析；内部负责数据库、知识库、简报和问答 |
| 评价、深度分析与成文分离 | 标题语义判断只负责候选路由，L2 形成统一评分，L3 形成深度材料，L4 只在已有 L3 中编排报告 |
| 单一排序口径 | L2、L3 和 L4 共用 `rank_score`，同事件合并后按分数和容量选取，不设置来源或分类配额 |
| 事实可追溯 | 二手来源的重要事实应保留原始链接；简报不得编造无法由原文核验的事实 |
| 质量优先 | 不为任何事件设置身份豁免，也不为均衡牺牲高分文章；来源和分类分布仅用于运营观察 |
| 最小数据模型 | 稳定查询字段结构化保存，类型差异内容进入 `analysis_detail` JSON，避免表字段无限扩张 |
| 可追溯 | 原文、来源、模型、Prompt 版本、入选文章和选题审计均可关联 |

## 2. 总体架构与部署边界

### 2.1 分区架构

| 区域 | 主要职责 | 主要依赖 | 明确不部署 |
|---|---|---|---|
| 亚马逊云单节点 | 固定源采集、站内发现、浏览器降级、清洗去重、L2/L3 分析、受控导入 | Python 3.12、FastAPI、Chromium/Playwright、本地文件、外部模型接口 | MySQL、Redis、内部知识库、APM |
| 公司内部应用节点 | 受控导入、MySQL 持久化、知识库文件生成与上传、L4 选题、L5 简报、RAG 问答 | Python 3.12、FastAPI、MySQL、内部模型、知识库 SDK/接口 | 外部网页采集组件 |
| 公司公共服务 | 定时调度和任务监控 | XXL-Job；也可改用本地 crontab | 不属于本系统部署范围 |

两个应用均保持单体服务形态。任务执行入口、数据源配置、采集、分析、模型编排和导入客户端均为服务内部模块，不拆成独立微服务。

### 2.2 逻辑分层

```text
L0 采集层       固定 RSS/HTML -> 标题/链接候选 -> 站内自动发现 -> 浏览器降级
      |
L1 候选层       每源最多100条 -> 标题语义路由/规则降级 -> 首批正文最多50篇 -> 去重/延迟池
      |
L2 基础分析层   正文充分性检查 -> 分类/五维统一评分 -> 覆盖补采
      |
L3 候选排序层   rank_score准入 -> 同主题聚合 -> 统一排序 -> 前50篇
      |
L3 深度分析层   高分代表文章的背景、问题、方案、影响和参考价值
      |
受控导入        批次校验 -> 文章/分析/洞察入库 -> 运营统计 -> 知识文件生成/上传
      |
L4 编排层       成功L3池 -> 跨批次主题聚合 -> 统一排序 -> 篇幅控制
      |
L5 成文层       按分类分段生成 -> 完整性校验 -> 固定结构组装 -> 草稿/知识库
      |
RAG 问答        知识检索 -> 上下文组装 -> 内部模型回答
```

### 2.3 网络与数据边界

- 亚马逊云只访问已配置公开域名、外部模型接口、可选搜索服务和内部受控导入地址。
- 站内发现和可选外部搜索结果都必须匹配已配置域名，不能把任意互联网结果直接送入分析链路。
- 内部侧不主动访问外部资讯网站。
- 外部公开文章、摘要、分析和引用地址可导入内部；内部问答、业务数据、知识库检索结果不得回传亚马逊云。
- 导入接口通过网络白名单、HTTPS 及认证令牌保护；令牌和模型密钥只通过环境变量提供。

## 3. 端到端处理流程

### 3.1 任务触发

采集和简报服务均支持三种调度模式：

- `xxljob`：服务启动时注册 Executor，由公司调度平台触发。
- `crontab`：服务不注册 XXL-Job，由本机 crontab 调用现有 HTTP 接口。
- `none`：只保留现有 HTTP 手动触发能力。

模式通过各自 `config/config.properties` 的 `scheduler.mode` 选择，不改变部署和启动方式。接口协议、请求示例和验证步骤以 [API接口清单与验证方案](../验证/API接口清单与验证方案.md) 为准，本设计不重复维护完整报文。

现有交互边界保持不变：

| 区域 | 接口或任务 | 设计职责 |
|---|---|---|
| AWS | `GET /health` | 服务存活检查 |
| AWS | `POST /api/v1/jobs/collect`、`aiRadarCollectJob` | 手动或调度触发 L0 至受控导入流程 |
| AWS | `POST /api/v1/jobs/health-check`、`aiRadarHealthCheckJob` | 检查数据源可达性 |
| 内部 | `POST /api/v1/radar/import` | 接收并持久化 AWS 结构化结果 |
| 内部 | `POST /api/v1/jobs/briefing`、`aiRadarBriefingJob` | 触发 L4/L5 周报、月报、季报或专题报告 |
| 内部 | `POST /api/v1/qa/ask` | 内部知识库 RAG 问答 |
| 内部 | `GET /api/v1/tasks`、`GET /api/v1/tasks/{task_id}` | 查询已有任务状态 |

为支持采集策略对比和分阶段排障，AWS 可提供“只采集”和“从本地采集批次开始分析”的验证入口。验证入口不得实现独立爬虫或独立 L2/L3 编排，只负责参数校验、本地批次读写和执行阶段选择；正式任务与验证任务必须共同调用 `CollectOrchestrator.collect_stage()` 和 `CollectOrchestrator.analyze_stage()`。两者差异只通过显式副作用开关表达：

| 执行模式 | 来源策略 | 历史去重 | 写已处理缓存 | 自动发现 | L3补抓正文 | 内部导入 |
|---|---|---|---|---|---|---|
| 正式完整任务 | 当前生产配置 | 开启 | 开启 | 按配置 | 开启 | 开启 |
| 验证只采集 | 指定对比策略 | 关闭 | 关闭 | 关闭 | 不执行 | 关闭 |
| 验证批次分析 | 继承本地批次 | 仅批内去重 | 关闭 | 关闭 | 关闭 | 关闭 |

验证采集结果必须完整落到 AWS 本地文件，再由分析入口按批次读取。这样既可以独立核对每个来源的候选数、正文可用率和失败原因，也保证验证所用候选池、L2、L3规则与正式运行一致；任何业务规则修改只允许发生在共享阶段实现中。

### 3.2 L0 固定源采集

数据源由 `aws/config/config.properties` 管理，内部数据库的 `ai_radar_source` 用于来源映射和追溯。每个源至少包含编码、名称、类型、主题提示、地址、域名、采集方式和启用状态。主题提示只用于候选路由，文章最终分类仍由 L2 判断；历史 `selection_role` 字段仅为兼容保留，不再参与 L3 或 L4 选题。

现有来源对国际厂商、基础研究和行业资讯覆盖较多，但中文工程实践、开发工具、安全治理和国内金融科技信号不足。V1.4 将以下内容作为候选来源方向，不在设计阶段直接承诺 URL、RSS 或选择器可用性：

| 补充方向 | 候选来源 | 主要补足内容 | 准入要求 |
|---|---|---|---|
| Agent 与开发工具 | Spring、GitHub、OpenAI 开发者内容、CSDN AI 频道 | Java/Spring AI、MCP、Agent SDK、AI 编程工具和工程实测 | 官方源优先；社区文章必须有可核验代码、配置或原始链接 |
| 企业安全与治理 | 工信部漏洞平台、腾讯云安全等 | AI 工具漏洞、Agent/MCP 安全、企业治理和国内合规 | 监管或厂商一手信息优先，安全结论必须可追溯 |
| 国内工程与产业 | 机器之心、企业 AI 专栏、财联社 AI 产业等 | 国内工程实践、产品落地、国产算力和产业变化 | 使用独立频道；通用门户必须语义过滤和事实交叉验证 |
| 金融 AI | 监管机构、金融科技公司和垂直媒体 | 投研、风控、合规、量化、客服及金融基础设施 | 仍作为九大方向之一，不改变全局报告为金融资讯报告 |

候选来源上线前必须完成可达性、时间字段、正文质量、robots/服务条款、重复率和连续三次试采验证。无法稳定取得正文、长期只转载或缺少原始出处的来源不得仅因“中文工程内容”而加入生产清单。

来源结构采用观察目标而非硬性凑数：工程实践类内容希望达到报告主题的 40% 至 60%，中文来源希望达到 40% 左右，基础研究和学术内容原则上不超过 20%。短周期内真实资讯不足时允许自然偏离，不能为了比例选择低价值文章。

采集方式：

| `fetch_method` | 处理方式 |
|---|---|
| `rss` | 请求 Feed 内容，使用 feedparser 容错解析，提取条目和可用正文；摘要型 Feed 可按来源配置补读同域详情页 |
| `web` | 读取列表页，根据选择器发现详情页，再抽取正文 |
| `api` | 预留给可直接返回结构化资讯的公开接口 |

RSS 和普通 HTML 请求统一使用浏览器 User-Agent，默认超时 20 秒。HTML 源可配置：

- `list_selector`：列表页文章链接选择器，多个候选用 `|` 分隔。
- `detail_content_selector`：详情页正文选择器。
- `max_pages`、`page_url_pattern`：分页范围。
- `next_page_selector`：没有固定页码规律时，从列表页发现受 `max_pages` 限制的下一页链接。
- `max_articles`：单源配置的详情读取保护值；任务运行时由候选池全局上限统一约束。
- `request_interval_seconds`：详情请求间隔。
- `include_keywords`、`exclude_keywords`：语义分类不可用时的降级过滤，以及通用门户的前置保护规则。
- `detail_content_reject_keywords`：排除页脚、关注引导等误命中的伪正文节点。

列表扫描和详情读取分离。系统在任务时间范围内尽量遍历列表边界，每源最多保留 100 条候选元数据；候选达到上限时必须在审计中视为可能截断，不能把 100 条误认为来源全部内容。

同一逻辑来源可以配置有序的端点变体。首选端点在时间范围内的候选数未达到其来源级最低量时，系统继续尝试后备端点，而不是因取得 1 篇文章即停止；后备端点均未满足最低量时，保留候选数最多的部分结果，并标记为 `partial`。该规则只改善单一来源的采集完整性，不参与 L2/L3 评分、最终选题或任何资讯的绿色通道。

### 3.3 浏览器降级采集

普通 HTTP 发生以下情况时，系统自动尝试 Playwright：

- 请求异常、状态码错误或超时。
- 页面可见正文低于配置阈值。
- 页面具有常见 JavaScript 空壳特征，普通解析无法获得实际内容。

降级过程使用 Playwright 驱动无头 Chromium 内核浏览器，单节点内串行执行，避免浏览器并发耗尽内存。浏览器可以使用 Playwright 下载的 Chromium，也可以通过 `browser_fallback.executable_path` 或 `PLAYWRIGHT_BROWSER_EXECUTABLE_PATH` 复用操作系统已安装的 Chromium、Chrome 或 Edge。浏览器仍使用原数据源的 URL、选择器和域名约束，不引入第二套数据模型。

浏览器渲染是 `fetch_method=web` 内部的失败降级路径，不是独立的第三种数据源采集方式。因此采集统计仍显示 RSS、WEB 两类；应通过浏览器降级日志和次数判断是否实际生效。HTTP 手动触发时，整个同步采集链路在工作线程执行，避免 Playwright Sync API 与 FastAPI 的 asyncio 事件循环冲突。若 Playwright 或可用浏览器均不存在，系统记录告警并保留普通采集结果，不影响服务启动和其他来源。

若同一来源的普通请求已超时，后续详情请求直接优先使用浏览器降级，避免每篇文章重复等待 HTTP 超时。若响应包含验证码或访问挑战页特征，系统记录挑战 URL 并在本批停止该来源后续正文请求，仅保留标题、链接和已有摘要；系统不使用 stealth、浏览器指纹伪装、登录态或其他绕过验证码的方式。访问挑战、HTTP 超时、浏览器尝试与成功次数、RSS Feed 条目数均进入来源级采集诊断。

### 3.4 自动 URL 发现

自动发现是固定源采集的补充，不替换固定采集。触发条件为：

- 某一级分类经 L2 后低于最小文章数；或
- 已采集结果中单一来源占比超过配置阈值。

默认发现顺序：

1. 读取已配置来源域名的 `robots.txt`，获取其中声明的 Sitemap。
2. 尝试标准 Sitemap 路径，并递归读取受限数量的 Sitemap 索引。
3. 按 `lastmod` 和任务时间范围选择近期文章 URL；没有 Sitemap 时，从来源入口执行限页、限深度的同域链接遍历。
4. 使用同一个 `WebCrawler` 抓取详情正文，再进入日期过滤、L1 去重和 L2 分析。

上述默认链路不依赖 Chromium、外部搜索服务或 API Key。普通请求已能取得列表和详情时，即使浏览器未安装也可以完成站内发现；只有 JavaScript 动态详情页需要浏览器降级。

第一阶段的重要限制：

- 用户无需预先判断“重要来源”或手工提供 URL。
- 发现范围只在已配置域名内扩展，因此不会自动信任未知网站。
- `discovery.enabled=true` 时默认执行站内发现，单任务通过来源数、Sitemap 数、页数、深度和 URL 数保护源站与单节点资源；这些上限不是 L2 或简报选题额度。
- 外部搜索 API 仅为可选补充。只有将模式改为 `hybrid` 并显式配置搜索服务时才使用 `SEARCH_API_KEY`。
- 每个任务最多补充一轮，避免自动发现递归扩张。

### 3.5 L1候选池、清洗与去重

L1 将不同采集方式统一为 `RawArticle`，并将网络处理拆成候选发现与正文读取两段：

1. 每源保留最多 100 条标题、链接、发布时间和可用列表摘要，先规范 URL 并排除历史已处理项。
2. 对混排来源的候选标题执行轻量模型批量语义分类，输出 `ai_related`、预测一级分类、资讯类型提示、置信度和简短理由；AI 垂直官方源可按配置跳过相关性淘汰，但仍执行标题去重和方向预测。
3. 语义分类只负责正文读取优先级，不直接产生 L2 正式分类或评分。模型调用失败、返回数量不一致或置信度过低时，回退到来源频道规则及 `include_keywords/exclude_keywords`，不得因分类服务不可用阻断采集。
4. 基于语义相关性、预测分类、标题信息量、摘要可用性和发布时间，首批最多选择 50 篇读取正文；模型发布等重要内容只能通过这些普通特征竞争，不能获得固定加分或强制入选。
5. 其余候选只保存元数据。首批 L2 后按分类覆盖缺口补读，每个不足分类最多补读 10 篇，所有候选使用相同补读上限。
6. 正文清洗后不足 100 字符的文章标记为内容不足，不进行正式评分；正文合格后计算 `content_hash` 并执行内容去重。
7. 只有 L2 成功后才写入 `data/interim/processed_hashes.json`，L2 失败文章可在后续批次重试。

标题级语义判断只用于路由和补采，不能直接形成正式评分。延迟候选写入 `data/candidate_pool/{batch_no}.json`，并保存语义分类结果与实际 L2 分类，便于回归误杀率和误放率；自动发现结果也必须经过相同的正文充分性和去重流程。

### 3.6 L2 基础分析

L2 输入包括文章标题、摘要或正文摘录、最多约 3500 字符的清洗正文样本，以及来源名称、类型、域名、默认分类、采集方式、发布时间、抓取时间和内容长度。没有原生摘要时由正文生成临时摘录；没有足够正文时不做正式评分。

L2 输出：

- 中文标题、中文摘要、原文语言和标准术语。
- 一级分类、子分类、资讯类型。
- `briefing_focus`：该资讯在简报中最值得表达的客观重点。
- `analysis_detail`：按资讯类型保存的可选结构化详情。
- 关键词、技术标签、涉及机构。
- 五个排序维度：券商技术岗位领域匹配度、趋势重要性、工程参考价值、技术深度和时效性。
- 统一 `rank_score` 和仅作材料质量提示的 `need_deep_analysis`。

技术文章可以在 `analysis_detail` 中表达解决的问题、变化内容和实现方式；模型发布、投融资、政策、观点等文章按自身类型保存适合的信息。缺少某个维度时允许为空，不强迫所有文章套用技术文章字段，也不为每类资讯无限增加数据库列。

L2 对首批正文和覆盖补采正文使用完全相同的分类、评分规则。来源身份不能升降任何评分。`score_org_relevance` 评价的是文章与国内头部券商技术岗位的领域匹配度：证券业务 AI、软件研发与 AI 工程、必须了解的重大 AI 事件三者任一命中即可获得相应分数，但普通“与 AI 有关”不能自动得到高分。评分理由写入 `analysis_detail`，不增加独立数据库列。

### 3.7 L3 候选排序与深度分析

同时满足以下基础条件的文章先进入 L3 候选池：

- `rank_score >= 6.0`；
- 文章正文、分类和评分信息完整。

候选池按以下顺序收敛：

1. 按中文标题、关键词、技术标签和机构名称聚合同一分类下的相似主题，同一事件优先保留 `rank_score` 最高的代表文章。
2. 所有主题按同一个 `rank_score` 降序排列；同分时依次比较领域匹配度、工程参考价值、趋势重要性、技术深度和时效性。
3. 单批最多选择 50 个主题进入 L3；超过容量的文章统一记录 `excluded_by_capacity`。
4. 不设置来源轮询、来源比例、分类比例、重大事件绿色通道或特定资讯类型豁免。来源与分类分布继续记录在审计数据中，但不改变选取结果。

所有合格 L2 结果仍然受控导入并可用于知识检索。未进入或未成功完成 L3 的文章不进入简报正文候选；所有 L3 失败统一记录原因，不设置事件专属补偿，也不能静默退化为 L2 摘要。

候选文章在配置要求全文时必须能够取得有效正文。L3 输出约 600 至 1200 个中文字符的深度洞察，详细程度高于 L2 摘要、明显短于原文，保留背景、核心事实、实现机制、关键数据、影响和适用边界。非技术资讯按产品、市场、投融资、监管或生态事实自然分析，不强行编造技术方案；“有 L3”表示有足够成文材料，不表示所有资讯都必须包含技术实现。

`need_deep_analysis` 仅保留为 L2 对材料充分性的提示，不再作为 L3 门禁。高 `rank_score` 的非技术资讯同样生成类型化 L3；正文缺失或无法取得仍作为技术失败处理，不能静默退化为 L2 摘要。

### 3.8 受控导入

AWS 将批次、文章、L2 分析、L3 洞察和阶段运营统计组装成一个导入请求。`operation_metrics` 为向后兼容的可选字段，不改变现有触发接口和任务参数。内部接口依次执行：

1. 校验批次和文章结构。
2. 以 `batch_no` 保证批次幂等，以 `url_hash` 保证文章幂等。
3. 将来源编码映射为内部 `source_id`。
4. 在事务中写入批次、运营统计、文章、分析和洞察。
5. 数据库提交成功后生成知识文件，并在知识库可用时上传。

知识库失败不回滚已经成功的 MySQL 事务。AWS 导入请求失败时按配置最多重试 2 次；最终失败的完整请求和元数据写入 `data/failed_imports/`，供人工使用现有导入接口重试。

## 4. 分类、评分与简报选题

### 4.1 一级分类

报告和 L2 使用统一一级分类：

1. **大模型基础技术**：预训练、模型架构、推理优化、微调、压缩、MoE 等。
2. **Agent与智能体**：智能体框架、工具调用、规划、记忆、多智能体和工程化。
3. **多模态技术**：图像、视频、语音、3D、多模态理解与融合。
4. **AI基础设施**：算力、芯片、分布式训练、推理服务、数据与存储、云和边缘。
5. **生成式AI应用**：代码、内容、企业应用、Copilot 和产品化创新。
6. **安全与伦理**：安全、对齐、可信 AI、版权、数据治理、监管和伦理。
7. **开源生态**：模型、框架、工具链、社区和开源商业生态。
8. **行业动态**：厂商发布、人才、产业合作、投融资并购和行业变化。
9. **AI在金融领域应用**：智能投研、风控、合规、客服、运营、营销、量化和金融基础设施应用。
10. **其他AI相关**：无法归入前述分类但仍有观察价值的内容。

金融应用与其他一级分类并列。系统不是“金融资讯报告”，也不会为所有文章打金融相关性分。医学、化学、物理等非金融垂直应用若没有可迁移的通用 AI 技术价值，归入“其他AI相关”，低价值内容可在 L2 后丢弃；无需为这些垂直行业建立额外子分类。

### 4.2 子分类与资讯类型

`sub_category` 回答“属于该技术方向的哪个细分主题”，例如推理优化、工具调用或智能风控。`info_type` 回答“这是什么性质的消息”，两者维度不同，不重复。

资讯类型候选包括：技术方案、模型发布、产品发布、开源项目、研究论文、工程实践、行业动态、投融资并购、政策监管、观点分析、案例实践和其他。

### 4.3 统一评分

L2 记录五项评分属性，全部进入唯一排序分：

| 属性 | 权重 | 评价内容 | 主要判断方式 |
|---|---:|---|---|
| 组织相关性 `org_relevance` | 35% | 与券商技术岗位关注领域的匹配程度 | 仅允许0、1、3、6、8、10六档；缺失为0，其他值向下归档 |
| 趋势重要性 `trend` | 30% | 对本报告关注范围的技术或产业变化等级 | 重大事件9-10；重要变化7-8；持续方向4-6；小众更新1-3；无价值0 |
| 工程参考价值 `engineering` | 20% | 是否包含可复用的架构、代码、配置、部署、选型、成本或实践信息 | LLM 基于正文判断 |
| 技术深度 `tech_depth` | 10% | 技术机制、实验、架构、性能数据和适用边界的完整程度 | LLM 基于正文判断 |
| 时效性 `timeliness` | 5% | 发布时间相对采集时间的新鲜程度 | 当日10；1-2日9；3-4日8；5-7日7；8-14日6；15-21日5；22-30日4；更早依次降至0 |

```text
rank_score =
    org_relevance * 35%
  + trend         * 30%
  + engineering   * 20%
  + tech_depth    * 10%
  + timeliness    * 5%

默认准入门槛：rank_score >= 6.0
```

组织相关性采用六档领域匹配标准：10分为直接服务证券业务的软件或 AI 技术；8分为 AI Coding、SDLC、测试、Agent/MCP、RAG、模型网关、权限治理、DevSecOps、可观测性和数据工程等核心软件研发与企业 AI 工程；6分为重要模型、开发框架、国内重大产业或监管和企业 AI 治理等应知信号；3分为迁移价值有限的通用教程、单一云厂商或一般跨行业案例；1分为无通用软件方法的纯硬件、具身智能、机器人、医学、化学和工业视觉；0分为无关、猎奇或低质量内容。银行、保险、支付等非证券金融案例通常不高于3分，有明确证券迁移价值时可为6分。

趋势重要性同样以本报告范围为基准，不能因为文章在医学、硬件或其他行业内部重要就直接获得高分。推理框架、模型服务和分布式推理属于软件工程关注范围；纯 GPU、芯片和 CUDA 硬件优化属于低匹配方向。AI 安全重点关注 Agent 权限、数据泄露、模型治理和开发供应链安全，普通社会事件和通用硬件安全不得高分。

来源名称、类型和原文链接用于事实追溯，不形成单独分数或门禁。简报生成只能引用已成功完成 L3 的材料；技术数据、结论和引文必须能回溯到原文或 L3 分析输入，避免把来源类型误当作文章事实质量。

时效性按文章发布时间与采集时间的差值由程序确定：当日10分，1-2日9分，3-4日8分，5-7日7分，8-14日6分，15-21日5分，22-30日4分，31-60日3分，61-90日2分，91-180日1分，更早或发布时间未知为0分。模型只接收该基础分，不根据文风猜测时效性。

### 4.4 L2、L3 与 L4 共用排序分

V1.4 取消 L2 `value_score`、L3 `l3_rank_score` 与 L4 `report_rank_score` 多套排序口径。V1.5 进一步取消来源和分类配额：L2 形成 `rank_score`，L3 与 L4 在同事件聚合后直接按该分数和容量选取；来源、分类和资讯类型分布只作为质量观察指标。

新批次必须写入 `score_org_relevance` 和 `rank_score`。`value_score` 仅作为兼容别名同步写入同一个 `rank_score` 值，旧数据读取时才允许在 `rank_score` 为空的情况下回退到 `value_score`；任何新批次不得重新计算旧五维平均分。默认门槛6.0是本次标注回归的初始值，后续使用相同标注集比较5.5、6.0和6.5后再校准，不能根据单篇文章临时改分。

### 4.5 同事件聚合

L4 根据标题词、关键词、技术标签和机构名称计算相似度，在同一一级分类内聚合同一事件。每个主题最多关联 3 篇文章：

- 排名最高的文章作为主来源。
- 其他来源作为事实补充或交叉佐证。
- 简报只生成一个主题，不因多个网站转载或报道同一事件而重复占位。

聚合是轻量、可解释的规则。选题审计文件保存主题与文章 ID，后续可根据真实样本调整相似度阈值。

### 4.6 L4 最终编排

默认目标主题数：

| 简报类型 | 目标主题数 |
|---|---:|
| 周报 | 8 |
| 月报 | 14 |
| 季报 | 18 |
| 专题 | 12 |

L4 的常规候选只来自报告周期内成功生成 L3 的文章，并执行以下统一规则：

- 同一事件先聚合，再选取有完整 L3、来源可核验且 `rank_score` 最高的文章作为主材料。
- 工程实践、技术方案、产品/模型发布和安全治理主题合计希望达到 50% 左右，但仅作为质量观察指标，不以低分内容凑比例。
- 全部主题按 `rank_score` 降序选取至目标主题数，不设置来源、分类、top-3、重大主题或特定类型豁免。
- 成功 L3 主题不足目标数时生成较短报告并记录缺口，不得使用只有 L2 摘要的文章静默补位。

L4 仍负责周报、月报和季报的跨批次聚合、同事件合并、目标主题数和最终引用关系，因此不能删除；它不再与 L3 使用不同排序公式，也不能在没有深度材料的全部 L2 中重新选题。

### 4.7 重要信息的统一评价

旗舰模型或平台发布、关键开源项目、国内重要 AI 监管信号和重大工程能力提升不使用独立识别标记参与路由，也不设置绿色通道。它们应因文章事实在 `trend`、`engineering`、`org_relevance` 等普通维度获得相应分数，并与其他资讯执行相同的正文要求、L3阈值、主题聚合和批次容量。

系统不保证任何指定事件必然进入简报。重要信息未入选时必须能从文章级审计中解释为评分不足、正文不足、同主题合并、容量限制或L3失败；设计改进应通过来源覆盖和评分回归完成，不能为单篇事件临时修改规则。

## 5. 简报生成与知识应用

### 5.1 L5 固定文章结构

L5 只接收 L4 入选主题、对应 L3 洞察和来源信息，不直接从全部文章中自行选题，也不使用 L2 摘要替代缺失洞察。固定 Markdown 结构为：

```text
# AI技术趋势雷达｜时间范围

导语

## 信息速览

## 本期有内容的一级分类
### 自然主题标题
按内容类型生成连续自然段
来源链接

## 信号源索引
```

正文规则：

- 分类章节只输出本期有入选主题的类别，按统一分类顺序排列。
- 同一类内容放在同一个一级章节下，同一事件只写一个主题。
- 信息速览中的每个主题都必须在正文出现，正文主题也必须进入信号源索引，三处数量和主题标识必须一致。
- 技术方案、工程实践、模型和产品发布通常生成 300 至 600 个中文字符，用自然段说明面对的问题、具体变化、实现机制、关键数据和适用边界，不显示纪要式字段标题。
- 非技术资讯通常生成 180 至 350 个中文字符，按实际类型说明产品范围、市场变化、生态关系、监管事实或业务变化，不强行套用技术三段式。
- 第二自然段可以客观说明对企业技术体系、工程建设或业务结合的参考意义和边界，但不替读者输出主观行动建议。
- 所有主题保留来源链接，文末提供信号源索引。

为避免月报或季报在单次生成中因输出上限压缩、遗漏主题，L5 按一级分类分段生成正文，再单独生成导语；信息速览和信号源索引直接由程序根据 L4 主题确定性生成，最后按固定顺序组装。分段模型只负责表达输入主题，不得改变 L4 选题、合并不同主题或自行增加资讯。

组装后执行确定性校验：

1. `L4 入选主题数 = 信息速览主题数 = 正文主题数 = 信号源索引主题数`。
2. 每个主题至少有一个来源链接；多来源主题保留主来源和补充来源。
3. 每个主题都有成功 L3；技术机制、数据和边界由分段 Prompt 约束并进入后续内容评审，不用 L2 摘要替代缺失 L3。
4. 程序校验分类标题、主题标记、来源链接、分类顺序和必需章节符合固定 Markdown 结构。
5. 校验失败时不把不完整内容标记为成功草稿；保存缺失主题和失败原因，供分段补生成或重新触发任务。

单次 L5 模型调用超时为 300 秒，重试次数为 0；分类分段是显式生成步骤，不属于同一请求的自动重试。只有组装和完整性校验成功后才写入 `ai_radar_briefing_draft.content` 并生成知识库 Markdown 文件。

### 5.2 英文资讯处理

英文文章不翻译全文：

- 原文正文作为核验和知识召回的主内容。
- L2 生成中文标题、中文摘要和标准术语，作为中文阅读入口和辅助索引。
- L3 和简报统一使用中文输出，英文技术名词保留原文或使用固定术语映射。
- 同一知识文件中可包含原文块、中文摘要块和术语映射信息，避免中文译法分散英文术语的精确召回。

### 5.3 知识文件类型

内部侧生成三类 Markdown：

| 类型 | 主要内容 | 粒度 |
|---|---|---|
| `article_summary` | 原文信息、中文摘要、分类、标签、评分和原文链接 | 每篇文章 |
| `deep_insight` | L3 背景、问题、方案、影响和参考价值 | 每篇高价值文章 |
| `briefing` | 经 L4/L5 生成的周报、月报、季报或专题报告 | 每期简报 |

建议知识库标签 Schema：`content_type`、标题、作者、来源、原文 URL、发布日期、一级分类、子分类、资讯类型、关键词、技术标签、机构、语言、统一排序分、模型和 Prompt 版本。知识库 SDK 当前若不支持上传时直接传标签，标签仍保存在 Markdown 元数据和 `ai_radar_kb_mapping.tags`，待平台能力开放后复用。

### 5.4 RAG 问答

问答接口在内部侧执行：校验访问令牌，检索知识库，按相关性组装上下文，再调用内部大模型生成带来源依据的回答。知识库不可用时返回明确失败，不将内部问题转发到外部模型。问答接口和请求格式不在本次改造范围内。

## 6. 数据与持久化设计

### 6.1 AWS 本地数据

AWS 不使用 MySQL 或 Redis，单节点本地目录承担短期状态和失败恢复：

| 路径 | 用途 |
|---|---|
| `data/interim/processed_hashes.json` | 已处理 URL/内容指纹，支持跨批次去重 |
| `data/failed_imports/*.payload.json` | 导入失败的完整请求体 |
| `data/failed_imports/*.meta.json` | 导入失败原因和任务统计 |
| `data/collection_audit/*.json` | 每批来源采集量、分析保留/丢弃和阶段统计 |
| `data/candidate_pool/*.json` | 延迟候选的标题、链接、语义相关性、预测分类、资讯类型提示和路由置信度 |
| `data/analysis_snapshots/latest.json` | 最近一次正式任务中已读取正文、可重新执行 L2/L3 的文章快照 |
| `data/source_health.json` | 各来源最近状态、连续空结果或失败情况 |
| `.playwright-browsers/` | 服务用户可访问的 Chromium 文件 |

中间结果仅用于失败重试、排障和审计，应配置目录权限、磁盘告警和定期清理策略。日志由应用写入本地文件，不接入 APM。

### 6.2 内部 MySQL

MySQL 保存 8 张核心表：

| 表 | 职责 | 关键约束 |
|---|---|---|
| `ai_radar_source` | 来源主数据和来源编码映射 | `source_code` 唯一 |
| `ai_radar_import_batch` | 导入批次、任务类型和成功失败统计 | `batch_no` 唯一 |
| `ai_radar_pipeline_operation` | 每批 L1/L2/L3 数量及来源、分类分布 | `batch_no`、`import_batch_id` 唯一 |
| `ai_radar_article` | 原始文章与清洗正文 | `url_hash` 唯一，索引发布时间和内容指纹 |
| `ai_radar_article_analysis` | L2 分类、类型、详情、排序维度和统一分数 | 每篇文章一条当前分析，索引分类、类型和排序分 |
| `ai_radar_deep_insight` | L3 深度洞察 | 每篇入选文章一条当前洞察 |
| `ai_radar_briefing_draft` | L5 简报正文、时间范围和引用关系 | 索引简报类型、时间范围和审核状态 |
| `ai_radar_kb_mapping` | 数据库对象与知识库文件映射 | `kb_type + kb_file_id` 唯一 |

精确字段、类型、索引和初始化来源以 [ai_radar_ddl.sql](../tech-research/sql/ai_radar_ddl.sql) 为唯一建库依据。该脚本用于直接重建数据库，后续结构更新均直接合并到该文件，不再新增常规增量 SQL 文件；设计文档不再复制整份 DDL，避免两个版本不一致。

稳定、常查询的 `category`、`sub_category`、`info_type` 保持独立字段；随资讯类型变化的内容及 `org_relevance_reason`、`trend_reason` 存入 `analysis_detail` JSON。`briefing_focus` 是跨类型通用的成文输入。来源表的 `selection_role` 仅为兼容保留且不参与选题；分析表保存 `score_org_relevance` 和 `rank_score`；`value_score` 仅为旧字段兼容，新批次写入与 `rank_score` 相同的值且不参与独立排序。金融应用不增加专属评分列或专属场景列。

### 6.3 候选与选题审计

AWS 每批在 `data/collection_audit/` 保存各来源候选数、是否触达 100 条上限、语义筛选结果、正文读取、L2、L3 和导入统计，并在 `data/candidate_pool/` 保存未读取正文的候选。文章级 L3 结果至少区分 `success`、`score_below_threshold`、`excluded_by_capacity`、`full_content_failed`、`llm_failed`、`parse_failed` 和 `merged_topic`，避免只能看到洞察缺失而无法解释原因。

L4 每次选题后将候选数、主题数、来源分布、分类分布、内容类型分布、关联文章 ID、洞察 ID 和排除原因保存到内部节点 `data/briefing_selections/`。L5 额外记录信息速览、正文和信号源索引的主题数量、缺失主题和校验状态。

内部 `ai_radar_pipeline_operation` 每个采集批次一行：L1 记录时间范围内候选总数、语义筛选和来源分布；L2 记录保留数量、来源和分类分布；L3 记录候选、成功洞察、来源/分类分布及失败原因汇总。补采、内容不足、重复、语义分类回退和 L3 结果原因等易扩展指标存入 `stage_detail` JSON。AWS 仍不连接 MySQL，该统计随现有受控导入请求由 internal 在同一事务中写入。

简报质量至少观察以下指标：常规入选主题 L3 覆盖率 100%、技术类候选 L3 成功率不低于 80%、正文主题覆盖率 100%、来源索引完整率 100%，以及来源、分类和内容类型实际分布。分布指标只用于发现采集或评分偏差，不作为强制选题门槛。

## 7. 配置、模型与运行控制

### 7.1 关键配置

| 配置域 | 主要参数 | 当前设计值 |
|---|---|---|
| L2 | 超时、并发、重试 | 120 秒、3、0 次重试 |
| L1 候选池 | 单源候选、首批正文、分类补采、最小正文 | 100、50、每不足分类最多10、100字符 |
| 标题语义路由 | 批量大小、超时、置信度、失败降级 | 开发时通过真实样本确定；失败回退来源规则和关键词 |
| L3 | `rank_score` 门槛、超时、并发、重试 | 6.0、300 秒、1、0 次重试 |
| L3 候选排序 | 批次容量、主题相似度、选择方式 | 50、0.34、同事件聚合后按分数降序 |
| 浏览器降级 | 启用、超时、最小正文、系统浏览器路径 | `true`、45 秒、300 字符、默认留空 |
| 站内发现 | 模式、来源数、页数、深度、单源 URL、分类最小量、来源集中阈值 | 默认启用；`site`；9；3；2；12；2 篇；40% |
| 外部搜索补充 | 启用、接口、查询次数 | 默认关闭；仅 `hybrid` 模式使用 |
| 导入 | 超时、重试、退避 | 现有配置控制，最多重试 2 次 |
| L4 | 成功 L3 池、统一排序分、主题数 | 见 4.6；不设置来源/分类配额，不允许 L2 静默补位 |
| L5 | 分类分段、超时、重试、完整性校验 | 每段 300 秒、0 次重试，四处主题数量必须一致 |

Prompt 配置：

- AWS L2：`aws/config/prompts/l2_summary.properties`
- AWS L3：`aws/config/prompts/l3_deep_insight.properties`
- 内部 L5：`internal/config/prompts/l5_briefing.properties`

V1.4 开发时新增标题语义路由 Prompt。它必须与 L2 Prompt 分离并单独版本化，避免低成本候选判断被误用为文章正式分析。

Prompt 文件必须保留版本号，分析结果保存实际模型和 Prompt 版本。分类、评分或成文规则变化时先更新 Prompt 版本，再通过真实样本回归。

### 7.2 环境变量

AWS 主要环境变量：`OPENAI_API_KEY`、`OPENAI_BASE_URL`、`IMPORT_ENDPOINT_URL`、`IMPORT_AUTH_TOKEN`、`PLAYWRIGHT_BROWSERS_PATH` 和 XXL-Job 配置。`SEARCH_API_KEY` 只在主动启用外部搜索补充时需要。

内部主要环境变量：MySQL 连接项、内部模型地址和凭证、知识库数据集配置、接口认证令牌和 XXL-Job 配置。

真实值只能写入 `secrets/.env` 或运行环境，`.env.example` 只保留占位符。日志、开发记录和设计文档不得记录真实密钥。

### 7.3 失败隔离与重试

- 单个来源采集失败不阻塞其他来源。
- 单篇 L2/L3 失败不阻塞同批其他文章；模型调用不自动重试，避免长时间重复占用。
- 正式任务在完成 L2/L3 后保存已读取正文快照。后续以 `scope=rerun` 触发时跳过外部采集、历史去重、标题路由和候选配额，直接使用最近快照重跑 L2/L3；快照不存在时明确失败。
- 重分析导入会覆盖相同 URL 的 L2 结果，并将本轮重分析文章中未重新生成的旧 L3 洞察标记为 `superseded`，使其不再参与后续简报。
- L3 候选选择、正文获取和模型调用分别记录结果原因；所有失败使用同一处理流程，简报不使用 L2 摘要补位。
- 浏览器降级只在普通抓取失败或正文不足时触发，不改变服务启动流程。
- 来源端点候选不足时尝试有序后备端点；均不足时保留最佳部分结果并标记 `partial`，不将其伪装为完整成功。
- 验证与正式采集均记录 Feed 条目数、候选下限、HTTP 超时、浏览器降级和验证码/访问挑战；验证码出现后本批熔断该来源详情请求，不绕过访问控制。
- robots.txt、Sitemap 或站内遍历不可用时只记录日志，不影响固定数据源任务；外部搜索默认不参与。
- 导入失败保存完整请求，可在网络或内部服务恢复后通过现有接口重放。
- 知识库不可达不影响内部服务启动，也不回滚 MySQL 入库；相关失败应记录映射状态。
- L5 分段或完整性校验失败不生成成功草稿，保留已有文章、洞察和缺失主题清单，后续可重新触发简报任务。

## 8. 部署与运维设计

### 8.1 AWS 单节点

实际部署目录为 `/app/001804/aws`，服务用户为 `appadmin`。沿用现有安装、启动和 systemd 方式，不新增独立爬虫进程。浏览器依赖安装步骤：

```bash
cd /app/001804/aws
source venv/bin/activate
pip install -r requirements.txt

export PLAYWRIGHT_BROWSERS_PATH=/app/001804/aws/.playwright-browsers
python -m playwright install chromium
```

若 Linux 缺少 Chromium 系统库，由具备系统权限的运维人员执行：

```bash
python -m playwright install-deps chromium
```

生产 systemd 单元通过 `Environment=PLAYWRIGHT_BROWSERS_PATH=/app/001804/aws/.playwright-browsers` 固定浏览器目录。安装后需确认目录及文件对 `appadmin` 可读和可执行。

若服务器已经安装系统 Chromium，可跳过浏览器下载，在 `.env` 或 systemd 中配置：

```bash
PLAYWRIGHT_BROWSER_EXECUTABLE_PATH=/usr/bin/chromium
```

Windows 功能调试也可直接复用 Edge，例如 `C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe`。执行文件路径只决定浏览器位置，不改变采集流程；运行服务的账号必须对该文件具有读取和执行权限。

### 8.2 内部节点

内部节点沿用现有 Python 虚拟环境、systemd 和 MySQL 连接配置。知识库 SDK `talentsview` 为可选能力：安装包可用时启用上传和检索；不可用或知识库地址不通时，服务及 MySQL 导入仍应可运行，知识库相关操作返回明确状态。

### 8.3 日志和监控

- 两侧应用日志均写本地文件，采用轮转和保留周期控制磁盘占用。
- 调度任务状态由 XXL-Job 或本地调度日志观察，不单独建设任务状态服务。
- AWS 重点监控来源成功率、空结果次数、语义分类回退、浏览器降级、L2 成功率、L3 各结果原因和导入失败文件数。
- 内部重点监控导入成功率、数据库连接、知识库上传、L4 成功 L3 候选分布、洞察覆盖率和 L5 主题完整性。
- `source_health.json` 用于识别长期失效源，但不自动把某个来源定义为重要或可信。

## 9. 安全与合规

1. 数据源必须使用公开可访问内容，并遵守网站服务条款、robots 约束、访问频率和版权要求。
2. HTML 与浏览器采集采用限页、限量和请求间隔，不绕过登录、验证码或付费墙。
3. 自动搜索只发现公开 URL，且受已配置域名白名单约束。
4. 外部模型只处理公开资讯，不发送公司内部数据。
5. 受控导入和内部问答接口必须认证，日志不得记录密钥、完整认证头或敏感业务输入。
6. MySQL、失败请求、审计文件和知识库内容按公司内部数据管理要求设置访问权限和保留周期。
7. 简报中的关键事实、数字和结论必须能够回溯到原文或分析结果；模型不得虚构来源和技术细节。

### 10.2 V1.4 实施边界与顺序

V1.4 的代码、SQL、配置和 Prompt 与本文同步交付。后续调整必须继续使用同一份人工标注回归集，避免在单篇文章驱动下临时改变全链路规则：

1. **数据与候选运营**：分批验证中文工程、安全和金融 AI 来源，持续观察标题路由召回率、误放率和正文成功率。
2. **评分回归**：使用标注样本监控六档领域匹配度和 `rank_score` 分布，比较5.5、6.0和6.5门槛；以前36篇14是/15否为基线，目标为前36篇至少25是且不超过8否。
3. **L3/L4质量**：观察文章级失败原因、成功 L3 覆盖和来源/分类分布，所有候选坚持同一准入与按分排序规则。
4. **L5质量**：监控分类分段生成、程序化组装和主题/来源完整性校验，失败时保留审计而不生成成功草稿。
5. **容量校准**：L3 单批容量按本次确认提高到50；继续记录单批耗时、L3成功率和月报候选使用率，容量变化不得重新引入来源或分类配额。

V1.4 不改变既有 HTTP、XXL-Job 和 crontab 的生产触发协议；验证文档已补充新增的分阶段采集与分析验证接口及其调用方式。L3失败不触发跨节点补偿接口，L4只消费已经成功入库的深度洞察。

以下能力仍在出现明确需求后再实施：扩大到未知域名的开放搜索、独立抓取队列或浏览器集群、跨期人工编辑工作台，以及知识库平台原生标签同步。
