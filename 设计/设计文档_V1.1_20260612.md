## 1. 总体部署边界

系统分为两个运行区：

| 区域 | 定位 | 主要职责 |
|---|---|---|
| 亚马逊云侧外部公开资讯处理区 | 面向公网资讯采集与公开内容分析 | 访问外部技术网站/RSS/API，完成采集、解析、去重、摘要、分类、深度分析、结构化结果生成 |
| 公司内部知识库使用区 | 面向内部知识沉淀、检索和问答 | 接收亚马逊云侧加工成果，写入内部数据库和知识库，基于内部大模型提供检索和 RAG 问答 |

**边界原则**：

- 亚马逊云侧只处理外部公开资讯，不处理公司内部业务数据
- 外部大模型只接收公开文章内容和分析提示词
- 公司员工问题、内部知识库内容、内部业务数据不发送至亚马逊云侧或外部大模型
- 内部检索、问答、知识复用全部在公司内部完成
- 亚马逊云侧仅通过受控导入接口进入内部，不直连内部数据库
- 亚马逊云单节点部署，日志本地存储，不接入 APM

## 2. 系统总体架构与模块划分

系统按"目标态架构、首期轻量实现"设计，分六层：

| 层级 | 部署位置 | 主要模块 |
|---|---|---|
| 数据源接入层 | 亚马逊云侧 | 数据源配置、白名单访问、RSS/API/网页采集 |
| 外部内容处理层 | 亚马逊云侧 | 内容解析、去重、有效性检查、外部大模型分析编排 |
| 结构化成果生成层 | 亚马逊云侧 | 文章记录、分析结果、报告草稿、任务元数据打包 |
| 受控导入层 | 公司内部侧 | 内部受控导入接口、数据校验、入库适配 |
| 内部知识服务层 | 公司内部侧 | 业务数据库、内部知识库、内部大模型、RAG 问答 |
| 应用输出层 | 公司内部侧 | 知识库检索入口、智能问答、周报/月报/专题报告草稿 |

运行治理能力（任务调度、日志、失败重试、数据源健康检查）贯穿两侧。

**首期约束**：不建设复杂前端管理后台；公司公用任务调度服务触发亚马逊云侧任务；不依赖 Redis、APM、多节点部署。

## 3. 数据流转与接口链路

| 链路 | 流向 | 说明 |
|---|---|---|
| 任务触发链路 | 公司公用任务调度服务 → 亚马逊云采集分析服务 | 按定时策略触发采集任务，支持补跑和范围参数 |
| 外部采集分析链路 | 亚马逊云采集分析服务 → 外部资讯源 / 外部大模型 | 访问公开网站、RSS、API，调用外部大模型分析公开文章 |
| 受控导入链路 | 亚马逊云采集分析服务 → 公司内部受控导入接口 | 将结构化成果、来源链接、任务元数据导入内部应用节点 |
| 内部知识使用链路 | 内部用户 → 内部知识库 / 内部大模型 | 内部检索、问答、简报复用均在公司内部完成 |

**安全边界**：公司业务数据、员工问题、内部知识库内容不得发送至亚马逊云侧或外部大模型；外部大模型只处理外部公开文章内容；进入公司内部的是公开资讯加工成果及任务元数据。

## 5. 内部 MySQL 表结构设计

数据库为公司内部 MySQL，不部署在亚马逊云侧。亚马逊云侧通过受控导入接口提交数据，由内部应用节点写入。

### 5.1 `ai_radar_source` — 数据源主数据

记录外部资讯来源，用于追溯文章来源和统计数据源质量。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | bigint PK AUTO_INCREMENT | 主键 |
| `source_code` | varchar(64) NOT NULL | 数据源编码，唯一 |
| `source_name` | varchar(128) NOT NULL | 数据源名称 |
| `source_type` | varchar(64) | 技术媒体、厂商博客、学术平台、开源社区等 |
| `access_url` | varchar(512) | RSS/API/网页地址 |
| `domain` | varchar(128) | 访问域名 |
| `enabled` | tinyint DEFAULT 1 | 是否启用 |
| `created_at` | datetime DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| `updated_at` | datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引**：`UNIQUE(source_code)`

### 5.2 `ai_radar_article` — 外部文章基础信息

记录采集到的外部文章基础信息。幂等去重依赖 `url_hash` 和 `content_hash`。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | bigint PK AUTO_INCREMENT | 主键 |
| `source_id` | bigint NOT NULL | 来源 ID，关联 `ai_radar_source.id` |
| `title` | varchar(512) NOT NULL | 原始标题 |
| `url` | varchar(1024) NOT NULL | 原文链接 |
| `url_hash` | char(64) NOT NULL | URL 哈希（SHA-256），唯一 |
| `author` | varchar(128) | 作者，可为空 |
| `publish_time` | datetime | 发布时间 |
| `crawl_time` | datetime | 抓取时间 |
| `raw_summary` | text | 原文摘要或抓取摘要 |
| `full_content` | longtext NULL | 文章完整原文，仅对需要精确解析的文章填充 |
| `content_hash` | char(64) | 内容指纹（SHA-256） |
| `import_batch_id` | bigint NOT NULL | 导入批次，关联 `ai_radar_import_batch.id` |
| `created_at` | datetime DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| `updated_at` | datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引**：`UNIQUE(url_hash)`，`INDEX(source_id)`，`INDEX(import_batch_id)`，`INDEX(publish_time)`

| `category` | varchar(128) | 资讯一级分类 |
| `sub_category` | varchar(128) | 资讯子分类 |
| `info_type` | varchar(64) | 资讯类型，如技术方案、模型发布、行业动态、投融资并购 |
| `briefing_focus` | text | 简报表达重点 |
| `analysis_detail` | json | 按资讯类型存放的结构化分析详情 |
| `keywords` | varchar(512) | 关键词，逗号分隔 |
| `tech_tags` | json | 技术标签数组 |
| `companies` | json | 涉及厂商数组 |
| `score_tech_depth` | decimal(3,1) | 技术深度评分 |

| `score_timeliness` | decimal(3,1) | 时效性评分 |

### 5.8 表设计约束

- `ai_radar_article` 不在 MySQL 中存储除精确解析外的全部文章全文，仅 L3 触发时填充 `full_content`
- `ai_radar_briefing_draft` 不存储正文，正文仅进入内部知识库
- 如后续知识库平台返回字段变化，可扩展 `ai_radar_kb_mapping` 字段

## 6. 知识库入库设计

### 6.1 入库粒度

三类内容均以独立 Markdown 文件入库：

| 类型 | 对应 MySQL 表 | 入库条件 |

| 标签字段 | 对应 MySQL 字段 |
|---|---|
| `briefing_type` | `ai_radar_briefing_draft.briefing_type` |

### 6.4 英文文章处理策略

英文来源文章不做全文翻译。全文翻译会显著增加 Token 成本和处理耗时，也容易造成技术术语多译、误译，降低后续检索和溯源质量。

系统采用“原文保留 + 中文入口 + 术语映射”的处理方式：

| 层次 | 内容 | 作用 |
|---|---|---|
| 英文原文块 | `raw_summary`、`full_content`、原始标题、原文链接 | 作为技术事实源和知识库主召回内容，保留 RLHF、LoRA、Attention 等英文术语 |
| 中文入口层 | L2 生成的 `title_cn`、`summary_cn`、中文关键词 | 供中文用户快速扫读，并支持中文查询入口 |
| 术语映射层 | `standard_terms`，含英文术语、缩写、中文标准译法、别名 | 建立中英文召回桥梁，保证同一术语输出一致 |
| 中文洞察层 | L3 基于原文全文生成中文深度分析 | 用于内部简报、知识沉淀和技术研判 |

处理约束：

- L2 仅读取公开文章标题、摘要或必要正文片段，输出中文标题、中文摘要、分类、关键词、技术标签和标准术语，不输出全文翻译
- L3 对高价值文章读取原文全文，输出中文深度分析报告，不改变或覆盖原始正文
- 知识库入库时同一 Markdown 文件包含英文原文块、中文摘要块和术语映射块；检索侧采用全文、关键词、标签和向量的混合召回，不仅依赖 embedding
- 原文链接必须保留，供用户在需要时回到公开原文核验上下文

### 6.5 来源引用

每份知识库文件头部统一携带原文链接，供内部用户点击打开。内部网络可能无法直接访问外部原文链接，该情况不影响系统功能。

### 6.6 上传机制（TalentsView SDK）

知识库上传使用 TalentsView Python SDK（`talentsview==1.2.2`）的 `KnowledgeBaseClient`，通过 `upload_chunks()` 方法将 Markdown 内容分片上传至 EIPLite 知识库平台。

**认证方式**：通过环境变量注入 SDK 所需凭证：

| 环境变量 | 说明 |
|---|---|

| 内部服务不可用 | 503 | 重试 2 次（10s / 30s 退避），总耗时不超过 1 分钟 |

## 9. 任务调度与执行策略

### 9.1 调度平台

使用公司 XXL-Job 分布式任务调度平台。亚马逊云侧以 XXL-Job Executor 模式接入，注册 JobHandler 到公司 XXL-Job Admin 服务端。调度平台配置定时策略后，按 cron 表达式自动触发 JobHandler，调度参数以 JSON 传入。

**XXL-Job Executor 配置**（`config.properties`）：

```properties
xxl_job.admin_addresses=http://<xxl-job-admin>/xxl-job-admin
xxl_job.app_name=ai-radar-executor
xxl_job.access_token=
xxl_job.executor_port=9999
xxl_job.log_path=/opt/ai-radar/logs/xxl-job
```

### 9.2 任务类型

| 任务 | JobHandler | 触发方式 | 调度参数 |
|---|---|---|---|
| 健康检查 | `aiRadarHealthCheckJob` | 定时（每 30 分钟） | 无 |
| 定时采集分析 | `aiRadarCollectJob` | 定时（每日 1-2 次） | `{"scope":"all"}` |
| 手动补跑（指定数据源） | `aiRadarCollectJob` | 手动触发 | `{"scope":"sources","sources":["aws-ml-blog","arxiv-cs-ai"]}` |
| 手动补跑（指定时间范围） | `aiRadarCollectJob` | 手动触发 | `{"scope":"timerange","from":"2026-06-01","to":"2026-06-10"}` |
| 失败批次重跑 | `aiRadarCollectJob` | 手动触发 | `{"scope":"rerun","batch_no":"IMP-20260610-001"}` |
| 简报生成 | `aiRadarBriefingJob` | 定时/手动 | `{"briefing_type":"weekly"}` |

**调度参数说明**：
- `from` / `to`：限定采集文章发布时间的范围，用于补跑遗漏时段
- `sources`：逗号分隔的数据源编码，对应 `ai_radar_source.source_code`，如 `aws-ml-blog`
- `batch_no`：失败批次号，用于重新处理未成功导入的批次

### 9.3 单次采集分析流程

```
XXL-Job Admin 触发 aiRadarCollectJob
         → Python JobHandler 接收调度参数
         → 读取配置 → 遍历数据源采集
         → 全部采集完成后统一解析
         → 去重校验（跨数据源）
         → 基础分析（L2，并发 ≤ 3）
         → 深度分析触发判断（value_score >= 8 且有 full_content，并发 ≤ 1）
         → 打包 → 受控导入
         → 记录日志
         → 返回执行结果给 XXL-Job Admin
```

### 9.4 失败处理

| 阶段 | 失败策略 | 影响范围 |
|---|---|---|
| 单个数据源采集失败 | 记录日志，继续下一源 | 不影响其他源 |
| 全部数据源采集失败 | 标记批次 failed，下个定时周期重试 | 整批次延后 |
| 单篇文章解析失败 | 跳过该篇，记录 URL + 原因 | 不阻塞其他文章 |
| 单次模型调用失败 | 重试 2 次（10s/30s），仍失败则跳过该篇并记录 `analysis_status = 'failed'` | 不阻塞其他文章 |
| 模型返回非 JSON | 重试 1 次（更严格 JSON 约束 prompt），仍失败则跳过 | 单篇 |
| 受控导入 503 | 重试 2 次（10s/30s），全失败则批次标记 failed，本地保留中间结果，下周期补传 | 整批次 |
| 模型 API Key 失效 | 跳过后续文章，记录 error.log，等待人工巡检 | 阻断 |

去重索引（`url_hash`）保证补跑不产生重复文章。

### 9.5 告警

| 场景 | 方式 |
|---|---|
| 连续 3 次定时采集全失败 | XXL-Job Admin 检测并通知 |
| 单个数据源连续 3 天不可达 | 健康检查日志异常，人工巡检 |
| 模型 API Key 失效或额度耗尽 | error.log 中体现，人工巡检 |
| 导入接口 3 次重试全失败 | error.log 明确记录，XXL-Job Admin 检测 |

不建独立的告警通道，依赖 XXL-Job Admin 状态检测 + 日志人工巡检。
## 10. 模型调用与 Prompt 管理

### 10.1 各阶段模型分工

| 阶段 | 部署侧 | 模型 | 输入 | 输出 |
|---|---|---|---|---|
| L2 基础分析 | 亚马逊云侧 | 外部大模型 | 公开文章标题 + 摘要/正文 + 分析 prompt | 中文标题、中文摘要、分类、关键词、标准术语、评分 |
| L3 深度洞察 | 亚马逊云侧 | 外部大模型 | 公开文章全文 + 深度分析 prompt | 中文技术背景、核心问题、方案、影响评估 |
| L5 简报生成 | 公司内部侧 | 内部大模型 | 已入库文章摘要 + 洞察 + 简报 prompt | 周报/月报/专题报告草稿 |
| RAG 问答 | 公司内部侧 | 内部大模型 | 用户问题 + 知识库召回片段 | 带来源引用的回答 |

### 10.2 分层模型策略

| 层级 | 适用阶段 | 说明 |
|---|---|---|
| 轻量模型 | L2 摘要分类评分 | 结构化输出不复杂，成本优先。如 GPT-4o-mini / Claude Haiku / Gemini Flash |
| 高质量模型 | L3 深度洞察 | 技术方案还原和影响评估需要强推理，日均触发量小 |

可通过 `config/config.properties` 分别配置 L2 和 L3 使用的模型。

### 10.3 Prompt 版本管理

Prompt 存储在 `config/prompts/` 目录下，每个文件包含 `version` 字段。每次修改 prompt 后更新版本号。每条分析记录在 MySQL 中落盘 `prompt_version` 和 `model_name`。同一篇文章用新版 prompt 重新分析时，通过 `article_url_hash` + `prompt_version` 判断是更新还是跳过。

### 10.4 分类与评分计算

L2 输出采用“资讯一级分类 + 子分类 + 资讯类型”的通用结构。`AI在金融领域应用` 是一级分类之一，与大模型基础技术、Agent 与智能体、多模态技术等并列，不作为所有文章的全局应用标签。

一级分类候选：

| 一级分类 | 说明 |
|---|---|
| 大模型基础技术 | 模型架构、训练、后训练、推理优化、评测、模型压缩等 |
| Agent与智能体 | 智能体框架、多智能体协作、工具调用、规划推理、记忆系统等 |
| 多模态技术 | 图像、视频、语音、3D、多模态理解与生成 |
| AI基础设施 | 算力、芯片、分布式训练、模型服务、MLOps、数据与存储 |
| 生成式AI应用 | 代码生成、内容创作、Copilot、企业级应用和通用产品化应用 |
| 安全与伦理 | AI 安全、对齐、隐私、版权、监管、可信治理 |
| 开源生态 | 开源模型、工具链、社区动态、开源商业化 |
| 行业动态 | 大厂发布、融资并购、人才趋势、产业案例和生态变化 |
| AI在金融领域应用 | AI 在投研、风控、合规、客服、运营、财富管理等金融场景中的应用 |
| 其他AI相关 | 不适合上述分类但仍与 AI 相关的信息 |

分类判定规则：

| 场景 | `category` | `sub_category` 示例 | 说明 |
|---|---|---|---|
| AI 在投研、风控、合规、客服、运营、财富管理等金融业务中的应用 | AI在金融领域应用 | 金融智能投研 / 金融风控反欺诈 / 金融合规监管科技 / 金融客户服务与运营 | 可通过 `category = 'AI在金融领域应用'` 精确筛选金融相关文章 |
| AI 在医学、化学、物理学、教育、制造、政务等非金融垂直行业中的应用 | 其他AI相关；若包含可泛化 AI 技术价值，则按底层技术归类 | 其他；或按底层技术选择已有子分类 | 不新增非金融垂直行业子分类。纯场景应用低优先级处理，不进入金融分类 |
| 仅出现金融机构、金融市场、投融资主体，但文章核心不是 AI 在金融业务中的应用 | 行业动态 / 开源生态 / 其他AI相关 | 厂商战略与投融资 / 其他 | 不得因为出现“金融”字样就归入金融应用分类 |
| 底层 AI 技术对金融未来有潜在参考价值，但文章本身不是金融场景 | 大模型基础技术 / Agent与智能体 / AI基础设施等 | 按技术内容选择 | 不强行归入金融分类，避免整体雷达被金融相关性污染 |

代码层对 L2 返回的 `category` 做候选值归一化：不在一级分类候选列表中的自由文本不会直接落库；医学、化学、物理学等非金融垂直行业分类会归一到兜底分类，金融分类仅保留为明确金融业务应用文章的一级分类。这样查询 `category = 'AI在金融领域应用'` 时，可以稳定获得金融相关文章。

对于 L2 判断为 `category = '其他AI相关'` 且 `value_score <= 5` 的低价值非金融垂直场景应用，AWS 侧在导入前直接丢弃，不触发 L3，也不写入内部 MySQL 和知识库。

资讯类型用于描述内容性质，例如：技术方案、模型发布、产品发布、开源项目、研究论文、工程实践、行业动态、投融资并购、政策监管、观点分析、案例实践、其他。

五个维度各 1-10 分：技术深度、工程参考价值、趋势重要性、来源可信度、时效性。`value_score` = 五项平均，保留两位小数。L3 触发阈值 `value_score >= 8`，阈值写入 `config.properties` 可按需调整。行业动态、投融资、政策监管等非技术文章可以在趋势重要性、来源可信度、时效性上得分，但技术深度和工程参考价值按实际内容给分。

### 10.5 成本控制

- L2 优先使用轻量模型控制单篇成本
- L3 仅高价值触发，日均 1-5 篇
- 去重先行，已入库文章不重复分析（除非 prompt 版本更新）
- `app.log` 中累计估算调用量 × 单价，日志可见，不在代码层做硬上限截断。

### 10.6 LLM 调用客户端设计（统一封装）

内部侧和亚马逊云侧的大模型调用均采用同一模式：通过 `httpx` 直连 OpenAI 兼容的 `/chat/completions` 端点，不经过中间 SDK（如 LangChain）。此模式在基线工程 `baseline-builder` 中经 25+ 处调用点验证。

**调用方式**（`src/clients/llm.py` — `call_llm()`）：

```python
import httpx

def call_llm(messages, model, temperature, max_tokens, timeout=300):
    url = f"{LLM_BASE_URL}/chat/completions"
    payload = {
        "model": model,
        "messages": messages,
        "temperature": temperature,
        "max_tokens": max_tokens,
    }
    headers = {
        "Content-Type": "application/json",
        "Authorization": f"Bearer {LLM_API_KEY}",
        "userid": LLM_USER_ID,
    }
    with httpx.Client(timeout=timeout) as client:
        response = client.post(url, json=payload, headers=headers)
        response.raise_for_status()
        return response.json()["choices"][0]["message"]["content"]
```

| 组件 | 说明 |
|---|---|
| HTTP 客户端 | `httpx`，同步调用，3 次重试 + 随机 jitter |
| 端点 | `{LLM_BASE_URL}/chat/completions`（OpenAI 兼容） |
| 鉴权 | `Authorization: Bearer {LLM_API_KEY}` + `userid: {LLM_USER_ID}` |
| Prompt 管理 | Jinja2 模板引擎，`prompts/templates/*.txt`，`render_template(name, **vars)` |
| JSON 解析 | `parse_llm_json()`：优先 `json.loads`，失败则 `json-repair` 修复，再失败降级为正则提取 |
| 超时 | 默认 300s，L2 分析可缩短，L3 深析可延长 |

**内部侧配置**（`tech-research/internal/secrets/.env`）：

| 环境变量 | 值 |
|---|---|
| `LLM_BASE_URL` | `http://168.63.65.40:8090/llm-service/v1` |
| `LLM_API_KEY` | `001804@nhbes2u5ejkqab5zfsra` |
| `LLM_MODEL` | `saas-doubao-15-pro-32k` |
| `LLM_USER_ID` | `013148` |
| `LLM_TEMPERATURE` | `0.01` |
| `LLM_MAX_TOKENS` | `4096` |

**亚马逊云侧配置**（`secrets/.env`）：

| 环境变量 | 说明 |
|---|---|
| `LLM_BASE_URL` | 外部模型 API 端点（如 OpenAI / Claude） |

**重试策略**：3 次尝试，退避时间 0.1s / 0.3s / 1s，每次叠加 0-0.1s 随机 jitter，避免重试风暴。

## 11. 安全边界与合规控制

### 11.1 数据边界

| 数据 | 流向 | 约束 |
|---|---|---|
| 外部公开资讯 | 公网 → 亚马逊云侧 | 仅采集白名单内域名 |
| 公开文章内容 | 亚马逊云侧 → 外部大模型 | 仅包含公开文章文本及通用分析 prompt |
| 公开资讯加工成果 | 亚马逊云侧 → 公司内部 | 通过受控导入接口入境 |
| 内部知识库内容 | 不出境 | 不发送至亚马逊云侧或外部大模型 |
| 员工问题 / 检索请求 | 不出境 | 同上 |
| 内部业务数据 | 不进入本系统数据流 | — |

### 11.2 网络层安全

- 亚马逊云出站：仅开放数据源域名 + 模型 API 端点（防火墙白名单）
- 亚马逊云入站：仅接受调度平台 HTTP 请求
- 内部受控导入接口：仅接受亚马逊云固定 IP 入站，防火墙管控
- 外部模型 API 调用：HTTPS 加密传输
- 内部导入接口不设应用层认证，安全边界由网络侧防火墙负责

### 11.3 审计与排查

- 每次导入均有 `ai_radar_import_batch` 记录，包含文章数、成功数、失败数、错误摘要
- `ai_radar_kb_mapping` 记录每条业务数据到知识库文件的映射关系
- `app.log` 记录完整运行链路，`error.log` 仅含异常
- 日志中不输出 API Key（仅显示后 4 位），不记录完整文章正文

### 11.4 合规确认

| 事项 | 结果 |
|---|---|
| 外部公开技术网站访问白名单 | 不需要合规审批 |
| 受控导入链路（云 → 内网） | 不需要合规审批，需走网络安全防火墙流程 |
| 外部模型 API 调用（海外数据传输） | 不需要合规审批，需申请外部大模型额度 |

## 12. 内部应用节点架构

内部应用节点部署在公司内部侧，承接受控导入、知识库同步、简报生成和问答服务。

### 12.1 模块拆分

```
内部应用节点（公司内部）
├── 受控导入接口         POST /api/v1/radar/import
│   └── 校验 → 写 MySQL → 生成 Markdown → SDK upload_chunks 上传知识库 → 写 kb_mapping → 返回
├── 简报生成服务
│   ├── XXL-Job Handler: aiRadarBriefingJob
│   ├── 查 MySQL 近一周/月文章 + 洞察
│   ├── 调用内部大模型生成简报
│   └── 简报入库（MySQL + SDK 上传知识库）
├── 问答服务
│   └── POST /api/v1/qa/ask
│       └── SDK KnowledgeBaseClient.query() 检索 → 内部大模型生成 → 返回带来源引用的答案
└── 任务状态查询
    └── GET /api/v1/tasks/{task_id}
```

### 12.2 导入即同步策略

受控导入接口采用"导入即同步"模式：同一请求内完成 MySQL 写入 → Markdown 生成 → SDK upload_chunks 上传知识库 → kb_mapping 回写。保证事务一致性，避免异步同步带来的一致性问题。单批次几十篇文章的场景下，同步耗时完全可接受。

### 12.3 内部大模型使用边界

- 简报生成：`call_llm()` → httpx POST → `{LLM_BASE_URL}/chat/completions`，Prompt 由 Jinja2 模板渲染
- RAG 问答：`SDK query()` 检索知识库 → `call_llm()` 基于召回片段生成回答（详见 §10.6）
- 内部大模型 API Key / userid 由内部应用节点环境变量管理，不出现在亚马逊云侧

## 15. 简报生成流程（L5）

简报由内部应用节点生成，不经过亚马逊云侧。

### 15.1 触发方式

| 简报类型 | 触发 | 覆盖范围 |
|---|---|---|
| 周报 | 每周一 08:00（XXL-Job） | 过去 7 天入库文章 + 洞察 |
| 月报 | 每月 1 日 08:00（XXL-Job） | 过去 30 天 |
| 专题报告 | 手动触发 | 指定分类 + 时间段 |

内部侧注册 XXL-Job JobHandler：`aiRadarBriefingJob`，调度参数 `{"briefing_type":"weekly"}`。

### 15.2 生成流程

```
XXL-Job 触发 aiRadarBriefingJob
    → 查 MySQL：时间范围内 value_score 降序取前 15 篇 + 所有深度洞察
    → 拼接内容（每篇标题 + 摘要 + 洞察）
    → 调用内部大模型生成简报正文（Prompt 见第 14.4 节）
    → 输出：概览 + 重点趋势 + 推荐文章 + 值得关注
    → 写 ai_radar_briefing_draft（元数据，不含正文）
    → 生成 Markdown 文件 → SDK upload_chunks 上传知识库 → 写 ai_radar_kb_mapping
    → review_status = 'pending'，等待人工审核
```

### 15.3 失败策略

| 场景 | 处理 |
|---|---|
| 时间范围内无新文章 | 跳过本次简报，记录日志 |
| 内部大模型调用失败 | 重试 2 次（10s/30s），仍失败记录 error，人工触发补生成 |
| 简报输出非合法 JSON | 重试 1 次（更严格格式约束），仍失败记录 error |

## 16. RAG 问答流程

### 16.1 问答链路

```
内部用户/系统 → POST /api/v1/qa/ask
    → 内部应用节点接收问题
    → SDK KnowledgeBaseClient.query() 检索知识库
        → 标签过滤（metadata_filter）→ 混合检索 → 召回相关片段
    → call_llm() 将召回片段拼入 Prompt → httpx POST {LLM_BASE_URL}/chat/completions
    → parse_llm_json() 提取结构化回答 → 返回答案 + 来源引用
```

检索通过 SDK `KnowledgeBaseClient.query()` 完成。生成通过 `call_llm()`（`src/clients/llm.py`）走 OpenAI 兼容 `/chat/completions` 端点完成，不经过任何中间 SDK。SDK 负责知识库检索，内部应用节点负责编排和调用内部大模型，保持检索与生成的解耦。LLM 调用客户端设计详见 §10.6。

### 16.2 检索策略

| 方式 | 说明 |
|---|---|
| 向量检索 | 语义匹配，适合非结构化文本 |
| 全文检索 | 关键字匹配，适合精确术语查询 |
| 混合检索（推荐） | 向量 + 全文综合排序 |

默认使用混合检索，可通过 `tag_filters` 缩小召回范围。

### 16.3 失败降级

| 场景 | 处理 |
|---|---|
| SDK 检索超时 | 503，提示稍后重试 |
| 检索返回空 | 返回 "未找到相关内容，建议调整问题或扩大检索范围" |
| 内部大模型生成超时 | 降级仅返回检索片段原文 + 来源引用 |
| 内部大模型不可用 | 降级仅返回检索片段 + 来源引用 |
