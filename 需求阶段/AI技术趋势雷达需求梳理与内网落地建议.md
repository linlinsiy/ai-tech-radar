# AI技术趋势雷达需求梳理与内网落地建议

生成时间：2026-05-19

更新记录：

- 2026-05-20：根据《20260520_ai技术雷达行动步骤讨论_原文.docx》补充近期行动步骤、环境方案、知识库入库方案、模型选型和资产交接事项。

## 1. 背景与现状

当前已有一套在互联网环境运行的“AI技术趋势洞察雷达”原型系统。该系统通过 RSS/网页源抓取、语义聚类、模型打分、深度精读分析等流程，持续跟踪全球 AI 技术动态，并将结构化结果同步至火山知识库。

现有系统已具备较完整的外网侧自动化能力：

| 层级 | 能力 | 产出 |
| --- | --- | --- |
| L0 | 从技术媒体、技术博客、学术平台等数据源定时抓取文章 | 原始文章池 |
| L1 | 语义聚类、主题合并、热点统计 | 动态话题表 |
| L2 | 文章价值打分、摘要生成、技术分类 | 文章总表 |
| L3 | 对高价值文章进行深度精读和洞察提炼 | 深度洞察卡片 |

目前接入的数据源约 12 个，覆盖国内技术媒体、海外技术博客、行业研究机构、学术平台等类型。现有系统能够支撑技术简报、趋势追踪、知识检索等场景，但主要运行在个人互联网环境和外部知识库环境中，尚未形成企业内部稳定可用的工程化系统。

## 2. 核心痛点

### 2.1 外网能力与内网知识库之间缺少通道

现有自动化抓取与分析能力主要依赖互联网环境。企业内网受网络访问限制影响，难以直接访问大量外部技术站点、开源社区、海外博客和学术平台。

因此，如果系统完全部署在外网，虽然能解决信息采集和分析问题，但无法自然沉淀为企业内网知识库，也难以与内部智能体、内部知识检索、内部报告流转体系打通。

### 2.2 现有原型缺少工程化维护

现有系统主要由个人基于 OpenClaw/低代码或脚本方式快速搭建，已经验证了可行性，但仍存在以下不足：

- 抓取脚本、调度任务、异常重试、失败告警等工程能力不足。
- 数据源失效、RSS 不稳定、网页结构变化后缺少持续维护机制。
- 大模型调用成本、模型降级、调用失败兜底策略仍需优化。
- 代码质量、可观测性、可部署性、权限和运维机制需要进一步审视。

### 2.3 数据源质量需要长期运营

当前 12 个数据源能提供基础内容，但文章质量、更新频率、稳定性存在差异。后续需要建立数据源治理机制：

- 持续补充高质量 AI 技术源。
- 识别低质量或重复度高的数据源。
- 对失效源进行自动检测和替换。
- 区分新闻源、学术源、厂商技术源、社区讨论源、内部人工精选源等不同类型。

### 2.4 知识库效果依赖入库结构与检索策略

只是把文章摘要或洞察内容简单入库，难以保证后续检索效果。需求讨论中明确提到，知识库建设需要考虑：

- 入库字段结构。
- 分词和标签规则。
- 技术主题分类体系。
- 召回率和准确率优化。
- 与内部智能体问答体验结合。

### 2.5 现有半自动技术雷达与人工报告需要整合

架构侧已有半手工半自动的技术雷达机制，另有专家/人工精编类技术报告来源。后续不宜形成多套割裂的技术雷达，而应将自动采集、架构师补充、专家精选三类来源整合为统一的技术洞察底座。

## 3. 目标

建设一套面向技术部/架构体系的 AI 技术趋势雷达内部版本，稳定支撑技术趋势洞察、知识库沉淀、智能检索和周期性运营输出。

核心目标包括：

1. 建立稳定运行的 AI 技术信息采集、分析、沉淀流水线。
2. 将外部 AI 技术动态转化为企业内部可检索、可复用、可运营的知识资产。
3. 支撑日/周/月技术简报、专题洞察、领导汇报、技术社区推送等输出形式。
4. 支撑员工通过内部知识库或智能体提问，例如“最近 Agent 领域有什么新进展”“有没有大厂的新技术能解决某类工程问题”。
5. 整合现有自动化原型、架构侧技术雷达、专家人工报告，形成统一技术雷达机制。

## 4. 需求梳理

### 4.1 信息采集需求

| 需求项 | 说明 |
| --- | --- |
| 外部数据源采集 | 支持技术媒体、厂商博客、学术平台、开源社区、行业研究机构等来源 |
| 数据源配置化 | 支持新增、停用、调整抓取频率、配置源类型和权重 |
| 定时抓取 | 支持每天多次自动抓取，也支持手动触发 |
| 去重与增量 | 按 URL、标题、发布时间、语义相似度等维度进行去重 |
| 原文留存 | 保留标题、链接、发布时间、来源、摘要、正文、抓取时间等元信息 |
| 数据源健康检查 | 识别长期无更新、访问失败、解析失败、内容质量低的数据源 |

### 4.2 分析加工需求

| 需求项 | 说明 |
| --- | --- |
| 主题分类 | 按大模型基础技术、Agent、多模态、AI 基础设施、生成式 AI 应用、安全伦理、开源生态、行业动态等方向分类 |
| 事件聚类 | 将同一技术事件或同一产品发布的多篇报道聚合 |
| 热度计算 | 基于报道数量、来源质量、传播范围、时间窗口计算主题热度 |
| 价值打分 | 对文章进行 1-10 分价值评估，识别高价值内容 |
| 中文摘要 | 生成面向内部读者的中文技术摘要 |
| 深度洞察 | 对高分文章抽取技术原理、架构变化、应用场景、影响判断、可借鉴点 |
| 标签体系 | 生成技术标签、公司/产品标签、场景标签、成熟度标签 |

### 4.3 知识库建设需求

| 需求项 | 说明 |
| --- | --- |
| 结构化入库 | 按文章、事件、洞察卡片、简报等对象分层入库 |
| 内网知识库同步 | 将外部加工后的结果同步到企业内部可访问知识库 |
| 检索优化 | 支持按主题、时间、来源、热度、分数、标签等检索 |
| 智能体问答 | 基于知识库提供趋势问答、技术方案推荐、专题追踪等能力 |
| 权限控制 | 根据内部知识库权限体系控制访问范围 |
| 可追溯性 | 每条洞察保留原文链接、来源和生成时间，便于复核 |

### 4.4 简报与运营输出需求

| 输出类型 | 说明 |
| --- | --- |
| 日报 | 每日高价值文章、热点事件、快速摘要 |
| 周报 | 本周趋势主题、重点洞察、值得关注技术、架构侧建议 |
| 月报 | 趋势变化、技术成熟度判断、重点方向复盘 |
| 专题报告 | 围绕 Agent、多模态、AI Infra、模型推理优化等主题生成专题分析 |
| 社区推文 | 面向内部技术社区或公众号的可读性内容 |
| 领导汇报材料 | 支持面向管理层的趋势摘要和决策参考 |

### 4.5 人工协同需求

| 需求项 | 说明 |
| --- | --- |
| 架构师补充 | 支持架构师手工补充来源、观点、案例和评分 |
| 专家精选接入 | 支持人工精编报告作为一个数据源进入统一雷达 |
| 审核发布 | 对外推送或领导汇报前支持人工审核与编辑 |
| 反馈闭环 | 用户可对文章质量、摘要质量、检索结果进行反馈 |

### 4.6 工程化需求

| 需求项 | 说明 |
| --- | --- |
| 调度运行 | 支持稳定的定时任务和手动补跑 |
| 异常处理 | 支持抓取失败、模型失败、入库失败的重试和告警 |
| 成本控制 | 低价值内容使用轻量模型，高价值内容使用高质量模型 |
| 监控看板 | 监控抓取量、失败率、入库量、模型调用量、成本、数据源健康度 |
| 配置管理 | 数据源、模型、Prompt、阈值、分类体系可配置 |
| 审计追踪 | 记录每次任务输入、输出、模型版本、Prompt 版本 |

## 5. 可行实现思路

### 方案一：参考外网原型，在内网重建程序版本

该方案是在内网重新实现一套技术雷达系统，将现有外网原型的 L0-L3 思路、数据结构、Prompt 和分类体系迁移到内网。

**优点：**

- 系统、数据、知识库均在内部环境，便于权限管理和组织级使用。
- 与内部知识库、内部智能体、内部报告体系集成更自然。
- 便于按照企业标准进行工程化、监控、运维和安全管理。

**不足：**

- 内网无法直接访问大量外部技术网站，信息源覆盖会受明显限制。
- 如果不解决外部信息进入内网的问题，内网版本容易变成“无源之水”。
- 需要重新建设抓取、解析、调度、模型调用、知识库入库等工程能力。

**适用条件：**

- 企业已有合规外部数据接入机制。
- 内网大模型、知识库、向量检索能力可满足需求。
- 希望长期建设为正式内部系统。

### 方案二：构建外网至内网的信息通道，在内网完成知识库构建

该方案保留外网环境的信息采集优势，在外网完成抓取和初步加工，再通过合规通道将结构化数据同步到内网，由内网完成知识库建设、检索智能体和运营输出。

**优点：**

- 充分利用外网环境访问数据源的便利性。
- 可以复用现有原型系统能力，启动成本较低。
- 内网只接收结构化、清洗后的数据，降低内网抓取复杂度。
- 便于分阶段落地，先解决“内容进内网”，再优化知识库和智能体。

**不足：**

- 需要建立外网到内网的数据传输通道，并满足安全合规要求。
- 数据同步链路需要做校验、脱敏、审计和失败补偿。
- 外网侧仍需具备一定工程化运维能力。

**适用条件：**

- 外网采集能力短期难以完全迁入内网。
- 企业允许经过审批的数据摆渡或同步机制。
- 目标是尽快形成内网知识库和可用输出。

### 方案三：混合模式，外网采集加工 + 内网知识库与应用 + 人工精选补充

该方案是在方案二基础上进一步扩展，形成统一内容底座：

1. 外网自动化系统负责采集外部公开信息。
2. 内网系统负责知识库、智能体、报告生成、权限和运营。
3. 架构师半自动雷达、专家人工报告作为内部数据源接入。
4. 数据源、标签、评分、Prompt、发布内容通过统一配置和治理机制管理。

**优点：**

- 同时解决外部信息采集、内部知识沉淀和人工专家判断问题。
- 能整合现有多套雷达/报告机制，避免重复建设。
- 更适合发展为技术部统一技术雷达平台。

**不足：**

- 方案完整度更高，涉及外网、内网、人工流程和运营机制，治理复杂度也更高。
- 需要明确系统边界、责任分工和数据同步规范。

**适用条件：**

- 目标不是单点工具，而是部门级长期运行机制。
- 需要兼顾自动化效率和专家判断质量。

## 6. 推荐方案

建议采用“分阶段混合模式”：

> 短期复用外网自动化采集与分析能力，优先打通外网到内网的结构化数据通道；中期在内网建设统一知识库、检索智能体和简报生成能力；长期将外部自动采集、架构师补充、专家精选报告统一纳入技术雷达运营体系。

推荐理由：

1. 内网直接采集外部网站存在网络限制，短期完全内网化成本高、效果不确定。
2. 现有外网原型已经验证 L0-L3 流水线可行，应优先复用其数据源、Prompt、分类和打分经验。
3. 组织真正需要的是内网知识库和内部可用的智能体，而不是单纯复制外网工具。
4. 架构侧已有半自动雷达，专家侧也有人工报告，应统一整合为一个技术雷达机制。

## 9. 分阶段实施建议

### 阶段一：需求固化与外网原型资产盘点

目标：明确统一需求，盘点可复用资产。

建议工作：

- 梳理现有外网原型代码、数据源清单、Prompt、表结构、知识库配置。
- 梳理架构侧已有半自动技术雷达流程和数据模板。
- 梳理专家人工报告的数据格式、输出周期和可复用字段。
- 明确内部知识库平台能力边界，包括分词、标签、向量检索、API、权限。
- 明确数据安全要求，确认外网至内网同步方式。

阶段产出：

- 需求规格说明。
- 现有资产盘点表。
- 数据源清单与质量评估。
- 内外网数据同步方案初稿。

### 阶段二：最小可用版本 MVP

目标：先让“外部洞察进入内网并可检索”跑通。

建议工作：

- 复用外网原型完成 L0-L3 结构化输出。
- 输出标准 JSON/CSV/Markdown 数据包。
- 建立外网到内网的合规导入流程，可先采用人工审核文件导入，后续自动化。
- 在内网知识库中建立 AI 技术雷达知识空间。
- 将 L2 文章摘要和 L3 深度洞察卡片导入知识库。
- 建立基础检索和问答能力。
- 生成第一版周报模板。

阶段产出：

- 可导入内网的结构化数据包。
- 内网 AI 技术雷达知识库。
- 基础问答智能体。
- 第一版自动生成周报。

### 阶段三：工程化增强

目标：提升系统稳定性、可维护性和运营效率。

建议工作：

- 将抓取、分析、入库、简报生成拆分为可观测任务。
- 增加任务调度、失败重试、日志、告警。
- 建立数据源健康检查和质量评分机制。
- 建立 Prompt 版本管理和模型调用成本统计。
- 建立轻量模型与高质量模型的分层调用策略。
- 增加人工审核、编辑、发布流程。

阶段产出：

- 稳定调度任务。
- 运维监控看板。
- 数据源治理机制。
- Prompt 和模型调用治理机制。

### 阶段四：统一雷达运营体系

目标：形成部门级统一技术雷达机制。

建议工作：

- 接入架构师手工补充和专家精选报告。
- 建立统一分类、标签、评分和发布规范。
- 支持日报、周报、月报、专题报告多种输出。
- 支持内部社区、公众号、领导汇报等发布渠道。
- 建立用户反馈机制，持续优化检索和内容质量。

阶段产出：

- 统一 AI 技术雷达运营流程。
- 多角色协作机制。
- 多渠道技术洞察输出。
- 技术趋势知识资产长期沉淀。

## 10. 数据通道建议

外网到内网的数据通道建议按安全成熟度逐步演进。

### 10.1 初期：人工审核文件导入

外网系统定期生成结构化文件，如 JSON、CSV、Markdown、压缩包等，经人工审核后导入内网。

适合 MVP 阶段，优点是实现简单、合规风险较低、便于人工把关；缺点是自动化程度较低。

### 10.2 中期：受控文件摆渡

外网侧生成标准数据包，进入受控文件摆渡目录或系统，内网侧定时读取、校验、入库。

适合稳定运行阶段，优点是自动化程度较高，便于审计和追踪；缺点是需要基础设施支持。

### 10.3 后期：审批后的 API 同步

如果安全策略允许，可以建立受控 API，将外网加工后的结构化洞察同步到内网接入服务。

适合长期平台化阶段，优点是实时性和自动化最好；缺点是安全评审、权限控制和运维要求更高。

## 11. MVP 范围建议

建议首个内部版本控制范围，优先验证闭环。

### 必做

- 复用现有 12 个外部数据源。
- 每日定时抓取和分析。
- L1 事件聚类、L2 摘要评分、L3 深度洞察。
- 结构化导出数据包。
- 人工审核后导入内网知识库。
- 内网支持基础检索和趋势问答。
- 每周自动生成一份 AI 技术趋势周报草稿。

### 暂缓

- 完全自动外网到内网 API 打通。
- 大规模数据源扩展。
- 复杂权限体系。
- 多渠道自动发布。
- 精细化用户反馈闭环。
- 面向全员的正式产品化界面。

## 12. 风险与应对

| 风险 | 影响 | 建议应对 |
| --- | --- | --- |
| 外部数据源访问不稳定 | 抓取失败、内容缺失 | 建立源健康检查、重试、备用源 |
| 内外网通道审批复杂 | 影响上线周期 | MVP 先采用人工审核文件导入 |
| 知识库检索效果不佳 | 问答体验差 | 优化字段、标签、分块、摘要和召回策略 |
| 模型调用成本过高 | 运行成本不可控 | 轻量模型处理大多数文章，高价值文章再深度分析 |
| 内容质量参差不齐 | 简报可信度下降 | 建立评分阈值、来源权重和人工审核 |
| 多套雷达重复建设 | 资源浪费、口径不一致 | 统一分类体系、数据底座和发布机制 |
| 生成内容有误 | 决策参考风险 | 保留原文来源，关键报告人工复核 |

## 13. 待确认问题

1. 内部知识库最终选型是什么，是否支持 API 入库、标签检索、向量检索和权限控制？
2. 外网到内网是否已有合规数据摆渡机制，支持哪些文件格式和自动化方式？
3. 现有外网原型代码是否可以作为正式工程化改造基础，还是仅作为需求和 Prompt 参考？
4. 架构侧已有技术雷达的数据模板、字段和发布周期是什么？
5. 专家人工报告是否能提供结构化输入，还是只提供成稿文档？
6. AI 技术雷达的首批核心用户是谁，是架构师、技术管理者、研发人员，还是全员？
7. 周报、月报、专题报告是否需要人工审核后发布？
8. 是否允许在外网侧保存原文全文，还是只能保存摘要和结构化洞察？

## 14. 行动步骤

根据 2026-05-20 讨论，后续行动建议从“需求固化、资产交接、技术验证、概要设计、MVP 建设”五条线并行推进。

### 14.1 本周完成：形成第一版需求与概要方案

目标：先形成一版可讨论、可评审的文档，融合创新实验室侧和架构侧诉求。

行动项：

1. 基于现有系统说明、5 月 18 日需求讨论和 5 月 20 日行动步骤讨论，完善需求文档。
2. 明确系统要抓取哪些类型的文章、处理哪些字段、如何打标签、如何评分、如何生成技术简报、如何进入知识库。
3. 在需求文档基础上补充概要设计内容，包括部署环境、数据流转链路、外网到内网通道、知识库入库方式、大模型调用策略。
4. 梳理架构侧已有技术雷达需求，以及创新实验室侧已有原型能力，形成统一版本。
5. 如有必要，进一步了解数科侧是否也有类似诉求，判断是否纳入首期范围。

阶段产出：

- 第一版需求与概要设计文档。
- 需求范围、MVP 范围和待确认问题清单。
- 初步行动计划。

### 14.2 资产交接：复用现有外网原型能力

目标：把已经验证过的原型资产转化为正式设计和工程实现的输入。

行动项：

1. 由现有原型负责人提供外网原型代码、任务配置、数据源清单、Prompt、评分标准和当前表结构。
2. 重点分析现有表结构如何服务两个输出：技术简报生成和知识库入库。
3. 提取当前 L0 抓取、L1 聚类、L2 评分摘要、L3 精读洞察的处理逻辑。
4. 梳理现有评分 Prompt，明确文章价值判断依据，例如是否有工程实践价值、是否有技术干货、是否只是普通新闻。
5. 将现有字段映射到内部知识库友好的标准数据对象，如文章、事件、洞察卡片、简报等。

阶段产出：

- 原型资产盘点表。
- 表结构和字段说明。
- Prompt 与评分规则说明。
- 可复用能力与需重构能力清单。

### 14.3 技术验证一：运行环境与外部访问能力

目标：判断长期运行是否可以尽量摆脱个人外部账号和个人机器，转为企业可管理环境。

讨论形成的新判断是：长期形态不应强依赖个人外部生态；如果企业境外环境可以访问目标外部网站，则可考虑在企业环境中完成外部数据抓取。

行动项：

1. 调研企业境外云环境可用性，例如境外 AWS 环境是否可申请、是否能访问目标技术网站。
2. 与相关负责人确认境外环境申请、评审和发放流程。
3. 验证目标数据源在企业境外环境中的可访问性，包括海外技术博客、学术平台、开源社区等。
4. 判断是否需要将“境外抓取环境”作为小课题或微创新课题进行申请。
5. 如果境外环境可行，优先设计为“企业境外环境采集 + 内网知识库沉淀”的长期架构。

阶段产出：

- 外部数据源访问验证结果。
- 企业境外运行环境可行性结论。
- 环境申请路径和责任人。

### 14.4 技术验证二：内网知识库与定制化入库

目标：确认内部知识库是否能满足技术雷达的检索和智能体问答需求。

讨论中重点提到，知识库不是简单把数据灌进去，需要考虑分词、分段、向量化、标签、召回率和 Agent Prompt。太维当前界面能力可能偏弱，但可通过高代码/代码框架方式定制入库规则，底层可能基于 ES 或知识库检索引擎。

行动项：

1. 调研太维知识库现有能力，包括存储、ES 检索、向量检索、标签、分词、分段、API 和权限。
2. 重点了解是否支持通过代码方式定制入库流程、分词规则、文本分段、向量化和召回策略。
3. 对接内部知识库实践负责人，了解已有项目如何通过框架实现定制化入库。
4. 设计面向知识库友好的入库结构，避免只存整篇文章。
5. 验证典型问题的召回效果，例如“最近 Agent 有什么新进展”“有没有新技术能帮助解决某类工程问题”。
6. 设计入库后智能体 Prompt，包括趋势查询、技术方案参考、专题追踪等问答场景。

阶段产出：

- 内网知识库能力调研结论。
- 定制化入库方案。
- 知识库字段、分段、标签、召回策略建议。
- 首批问答场景和 Agent Prompt 草案。

### 14.5 技术验证三：大模型选型与成本效果对比

目标：选择适合长期运行的模型组合，兼顾英文内容处理效果、成本和企业可用性。

讨论中提到，现有原型使用 Gemini，主要优势是对英文网站和技术内容处理较好；但该任务本身复杂度不算极高，长期运行不一定必须依赖国外模型，可评估豆包、通义千问、DeepSeek、Kimi 等国内模型。

行动项：

1. 梳理大模型在系统中的主要任务：摘要、价值评分、分类打标、高价值文章深度分析、技术简报生成。
2. 设计模型对比样本集，覆盖中文媒体、英文厂商博客、学术论文摘要、开源社区动态等。
3. 对比候选模型在英文理解、技术摘要、评分稳定性、标签准确性、深度洞察质量上的效果。
4. 对比不同模型的调用成本、企业可用性、部署/调用方式和合规要求。
5. 设计分层模型策略：低价值或常规文章使用轻量模型，高价值文章和简报生成使用效果更好的模型。

阶段产出：

- 模型选型对比报告。
- 推荐模型组合。
- 分层调用和成本控制策略。

### 14.6 概要设计：明确部署态与数据流转链路

目标：从需求进入设计，明确系统到底部署在哪里、数据怎么流、知识库怎么接。

行动项：

1. 画清楚外部数据从采集、清洗、分析、导出、传输、入库、检索、简报生成的完整链路。
2. 设计两种部署态：
   - 过渡态：复用外网原型，导出结构化数据，通过人工审核或文件摆渡进入内网。
   - 目标态：企业境外环境负责外部采集与初步加工，内网环境负责知识库、智能体和简报运营。
3. 明确每个环节的输入、输出、存储位置、失败处理和审计要求。
4. 明确哪些能力在外网/境外环境运行，哪些能力在内网运行。
5. 明确数据进入内网前是否需要清洗、脱敏、格式校验和人工审核。

阶段产出：

- 概要设计文档。
- 部署架构图。
- 数据流转图。
- 内外网数据通道设计。

### 14.7 6 月初目标：形成一个可演示版本

目标：争取在 2026 年 6 月初形成一版可演示、可评审的 AI 技术雷达内部版本。

建议 MVP 范围：

1. 复用现有外网原型或企业境外环境，完成一批外部 AI 技术资讯抓取。
2. 完成摘要、评分、分类、标签和高价值文章洞察生成。
3. 按标准字段导出结构化数据。
4. 将一批数据导入内网知识库或测试知识库。
5. 验证基础检索和问答效果。
6. 生成一份 AI 技术趋势周报草稿。
7. 汇总验证问题，形成下一阶段工程化改造清单。

阶段产出：

- 一版可演示 MVP。
- 示例知识库数据。
- 示例技术周报。
- 问答验证样例。
- 工程化待办清单。
